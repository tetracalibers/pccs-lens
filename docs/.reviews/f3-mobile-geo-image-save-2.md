# レビュー: F3 幾何パターン画像保存ボタンの改良

- レビュー対象: `docs/ideas/f3-mobile-geo-image-save.md`
- レビュー日: 2026-03-15

---

## 既存レビューからの改善状況

既存レビュー（`docs/.reviews/f3-mobile-geo-image-save.md`）で「要修正」とされていた項目の取り込み状況を最初に確認します。

**取り込み済みの項目（以前は未記載だったが現在のアイデアメモに含まれている）**

- `navigator.canShare({ files: [file] })` による事前確認
- `new File([blob], 'pattern.png', { type: 'image/png' })` によるFileオブジェクト変換の手順
- iOS Safariのユーザーアクション制約への対処
- エラーハンドリング方針（AbortErrorは無視、それ以外はconsole.errorのみ）
- `disabled={!bauhausSvg || bauhausLoading}` によるローディング中の無効化
- `generatePngBlob` ユーティリティ関数への切り出し方針
- 判定条件の組み合わせ表
- 検証シナリオ表
- `navigator.share` のモック方針
- ヘルプテキストの設計意図の補足
- `window.matchMedia(...)` による実装方針

既存レビューで指摘されていた重大な問題の大部分が解消されています。以下では現在のアイデアメモを改めてゼロベースで評価します。

---

## 1. 情報の網羅性（Specification Completeness）

評価: ⚠️ 改善推奨

理由:

- **共有するファイル名に `themeId` が使われていない**: 既存の `downloadPng` では `${themeId}-bauhaus.png` / `${themeId}-geometric.png` というファイル名でダウンロードしている。アイデアメモでは `'pattern.png'` という固定ファイル名が記載されている。保存時と共有時でファイル名の命名規則を揃えるかどうかの判断が仕様に欠けている。

- **Web Share API の `title` パラメータの仕様が不完全**: `title: 'カラーパターン'` とあるが、バウハウスとジオメトリックで同じタイトルになる。ユーザーが共有シートで見たときにどちらのパターンかを識別できないため、タイトルをパターン種別ごとに変えるかどうかの判断が必要。

- **`generatePngBlob` の出力先（ユーティリティファイルパス）が未定義**: 「ユーティリティ関数として切り出す」と記載しているが、どのファイルに配置するかが示されていない。

- **Canvas の解像度が仕様に記載されていない**: 実装ファイルでは `canvas.width = 300; canvas.height = 300` で固定されている。共有時も同じ解像度を使うのかが明示されていない。

改善提案:

- 共有時のファイル名を `${themeId}-bauhaus.png` / `${themeId}-geometric.png` に揃えるか、固定にする場合はその理由を記載する。
- `navigator.share` の `title` パラメータをパターン種別ごとに変えるかどうか（例: `'バウハウスパターン'` / `'ジオメトリックパターン'`）を明記する。
- `generatePngBlob` の配置先ファイルパスを記載する。

---

## 2. 仕様の明確性（Clarity）

評価: ✅ 良好

理由:

- 判定条件の組み合わせ表が追加されており、端末ごとの期待挙動が明確に整理されている。
- iOS Safariのユーザーアクション制約の説明は技術的に正確かつ理由も含めて記述されており、実装者が迷わない水準に達している。
- `pointer: coarse` と `hover: none` の組み合わせでタッチ対応ノートPCを除外する設計意図が説明されており、判定ロジックの背景が理解できる。
- ヘルプテキストの常時表示という仕様と、その背景となる設計意図が対で記載されており、仕様と意図が分かれていて読みやすい。

改善提案:

- 判定条件の組み合わせ表にある「ペン入力端末（Surface等）」行の「ヘルプテキスト：表示（タッチ併用の場合）」という記載が条件付きになっている。`any-pointer: coarse` メディアクエリは「タッチ対応ならば表示する」という判定のため、ペン専用端末でタッチ非対応の場合は非表示になる。「タッチ非対応のペン入力端末では非表示、タッチ併用のペン入力端末では表示」と明確に2行に分けるか、表の注記として補足するとよい。

---

## 3. 仕様の一貫性（Consistency）

評価: ✅ 良好

理由:

- ボタン表示条件と `disabled` 条件が対応している。
- ヘルプテキストの表示条件（`any-pointer: coarse`）とボタン表示条件が異なることについて、設計意図の補足説明が追記されており、意図的な差異として説明できている。
- 「判定条件の注意点」のスコープがボタンラベルの判定・ヘルプテキストの表示判定の両方に適用されると明示されており、以前の曖昧さが解消されている。

改善提案:

- 検証シナリオ表のシナリオ5（SVG再生成中）でヘルプテキストの期待値が「—」（未定義）になっている。再生成中はヘルプテキストの表示状態が変わらない（常時表示）という仕様なら「表示（端末条件による）」と記載する方が正確。

---

## 4. 実装可能性（Implementability）

評価: ✅ 良好

理由:

- `window.matchMedia('(pointer: coarse) and (hover: none)')` での判定方針、`$derived` または `onMount` 内での評価方針が具体的に記述されており、Svelte特有の実装上の判断が示されている。
- `canvas.toBlob` のPromise化と `async/await` チェーンによるiOS Safari対応は、現行実装がコールバック方式であることと照らし合わせると、必要な変更点が明確に示されている。
- `generatePngBlob` への切り出しと保存・共有での再利用という方針は、現行の `downloadPng` 関数の構造と整合している。

改善提案:

- `$derived` と `onMount` の使い分け基準が「または」として並列に示されている。SvelteKitのSSR環境では `browser` フラグが必要になるため、既存コードの `if (!browser) return` パターンと整合する実装方針（例: `onMount` 内で `window.matchMedia` を評価し `$state` に格納する）を補足することが有益。

---

## 5. 検証可能性（Testability）

評価: ✅ 良好

理由:

- 検証シナリオ表が追加され、端末条件・期待ラベル・期待クリック挙動・ヘルプテキストの4軸で整理されている。
- `navigator.share` と `navigator.canShare` のモック方針（`vi.stubGlobal`）が明記されている。
- Playwrightでの `emulateMedia` 活用方針が示されている。

改善提案:

- 検証シナリオ表に「`pointer: fine`, `hover: none`（ペン専用端末）」のシナリオが含まれていない。判定条件表には「ペン入力端末（Surface等）」行があるが、検証シナリオに対応するケースがない。境界ケースとして追加すると仕様の抜け漏れ検証に役立つ。

---

## 6. 将来拡張性（Extensibility）

評価: ✅ 良好

理由:

- 機能検出（feature detection）ベースの判定方針は、将来ブラウザサポートが変わっても特例コードが増えにくい。
- `generatePngBlob` をユーティリティとして切り出す方針は、同じ生成ロジックを他のパターンや他のコンポーネントで再利用できるようにする。

改善提案:

- 将来的にパターンの種類が増えた場合、現状の仕様では `disabled` 条件や共有ロジックをパターンごとに記述する設計になっている。このロジックをカード単位のコンポーネントに閉じ込める設計も選択肢として言及しておくと、将来の拡張コストを低減できる。ただし現時点での変更対象は `GeoPatternSection.svelte` のみであることが明示されており、範囲が明確な点は良い。

---

## 7. 仕様リスク（未定義挙動）（Specification Risk）

評価: ⚠️ 改善推奨

理由:

- **`generatePngBlob` でBlobが `null` になった場合のハンドリングが共有フローに未記載**: 現行実装では `if (!pngBlob) return` でBlob生成失敗を黙って無視している。アイデアメモの共有フローでは `const blob = await generatePngBlob(svgString)` の結果がnullまたはundefinedになった場合に何をするかが記述されていない。`generatePngBlob` がnullを返した場合に `new File([null], ...)` を呼ぶとエラーになるため、呼び出し側での `null` チェックが必要。

- **`navigator.canShare` チェック通過後に `navigator.share` が失敗した場合のユーザーフィードバックが未定義**: `canShare` は `true` を返したが `share` の実行時にエラーになるケースは、エラーハンドリングの「それ以外のエラー：`console.error` でログ出力のみ」で対処されているとも読めるが、明示的にこのケースをカバーしているとは書かれていない。ユーザー向けのフィードバックを出さなくてよい理由を一言添えると意図が明確になる。

改善提案:

- `generatePngBlob` の戻り値型と、Blob生成に失敗した場合の振る舞いをユーティリティ関数の仕様として記載する（例: `Promise<Blob>` を返し、nullの場合はrejectする）。そうすることで、共有フローの `try/catch` または「それ以外のエラー」節が失敗ケースをカバーすることを明示できる。
- `navigator.share` が失敗した場合にユーザー通知を出さない理由（例: モバイルOSの共有シートが失敗した場合にはOS側のフィードバックがある、または頻度が低い）を一言補足する。

## 総合レビュー

### 重大な問題

前回レビュー（`docs/.reviews/f3-mobile-geo-image-save.md`）で指摘されていた重大な問題はすべて解消されています。現在のアイデアメモには実装を阻むレベルの問題はありません。

以下は軽微ながら実装前に確認しておくべき未定義点です。

- `generatePngBlob` の戻り値型（`Promise<Blob>` か `Promise<Blob | null>` か）が未定義であり、呼び出し側の `null` チェックが必要かどうかが実装者によって解釈が割れる。
- 共有時のファイル名（`'pattern.png'` 固定）と既存の保存時ファイル名（`${themeId}-bauhaus.png`）の命名規則の不一致が仕様として意図的かどうかが読み取れない。

### 改善提案（重要度順）

1. **`generatePngBlob` の戻り値型と失敗時の振る舞いを明記する**: `Promise<Blob>` として定義しnullの場合はrejectする設計か、`Promise<Blob | null>` として呼び出し側でnullチェックする設計かを示す。前者の設計であれば、共有フローの `try/catch` または「それ以外のエラー」節が失敗ケースをカバーすることを明示する。

2. **共有時のファイル名に `themeId` を組み込むかどうかを決める**: ダウンロード時と共有時でファイル名の命名規則を揃えるか、意図的に固定にする場合はその理由（例: ファイル名がユーザーに見えないため固定で問題ない）を一言記載する。

3. **検証シナリオに「`pointer: fine`, `hover: none`（ペン専用端末）」のケースを追加する**: 判定条件表には「ペン入力端末（Surface等）」行があるが、検証シナリオに対応するケースがない。

4. **`$derived` と `onMount` の使い分け基準、または推奨方針を明記する**: SvelteKitのSSR環境では `browser` フラグが必要になるため、既存コードの実装パターンと整合する方針を明示する。

5. **ペン入力端末行のヘルプテキスト記載を明確化する**: 「表示（タッチ併用の場合）」という条件付き記述を、タッチ非対応と併用の2行に分けるか注記を追加する。

### 良い点

- 前回レビューで指摘されていたすべての重大な問題が漏れなく取り込まれており、仕様の熟度が大幅に向上している。
- iOS Safariの制約に関する説明は、制約の内容・原因・対処方法・なぜ従来の実装では駄目なのかの4点がすべて1段落に収まっており、実装者が背景を理解した上で実装できる水準の記述になっている。
- 判定条件の組み合わせ表（実装の仕様）と検証シナリオ表（テストの基準）の2つの表が役割を分担している設計は、仕様書としての構成として優れている。
- `userAgent` や画面幅による判定を明示的に排除しCSSメディアクエリと機能検出の組み合わせで判定する方針は、UA文字列の変動やビューポートの多様化に左右されない堅牢な設計判断であり、長期的な保守性が高い。
