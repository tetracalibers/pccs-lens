# Output Contract

著者のスタイル分析で作成または更新する成果物について、共通の出力形式、品質基準、根拠の扱い、ファイル間の責務を定義する。

このドキュメントは、次の分析および統合処理で共通して適用する。

- `thinking-flow.md`
- `writing-style.md`
- `stylistic-quirks.md`
- `refine-style.md`

各分析固有の出力構成は、それぞれの分析プロンプトに従う。ただし、記述形式、確度、根拠、更新方法については、この契約を優先する。

## 目的

この契約の目的は、分析担当や実行時期が異なっても、次の状態を保つことである。

- 同じ意味のルールが異なる形式で乱立しない
- 根拠の強さと主張の強さが対応する
- 思考、構成、表現、修正傾向が混同されない
- 記事固有の特徴が著者全体の特徴として一般化されない
- 新しい記事を追加した際に、既存ガイドを差分更新できる
- 文章生成時に実行可能なルールとして利用できる
- 分析上の仮説と、本文から直接確認できる事実を区別できる

最終成果物は分析レポートではなく、文章生成および推敲時に参照できる実用的なガイドとする。

## 対象となる成果物

### `thinking-flow.md`

著者が文章を書く際に行っていると考えられる判断を扱う。

主な対象：

- 執筆時の問題認識
- 説明方法を選択する判断
- 読者の理解状態に関する想定
- 抽象と具体を切り替える条件
- 説明を補足、省略、展開する条件
- 次の話題へ進む判断
- 推敲時の判断

記事の見出し順や段落構成そのものは、主な対象にしない。

### `writing-style.md`

文章の構成として観察できる傾向を扱う。

主な対象：

- 記事全体の情報配置
- 導入から本題への移行
- セクションの分割と接続
- 段落の構造
- 一文の情報量
- 定義、理由、例、注意点の配置
- 記事タイプによる構成の変化

内面的な意図や頻出表現は、主な対象にしない。

### `stylistic-quirks.md`

文やフレーズのレベルで観察できる表現上の癖を扱う。

主な対象：

- 頻出する言い回し
- 文末表現
- 接続表現
- 語彙の選択
- 断定や推測の調整
- 記号や表記
- 読者との距離感
- あまり使用しない表現

文章全体の構成やGit差分は、主な対象にしない。

### `refine-style.md`

AI草稿と人間による編集との差分から確認できる修正傾向を扱う。

主な対象：

- 情報の追加と削除
- 説明順の変更
- 表現の書き換え
- 段落や見出しの変更
- AIらしい定型表現の除去
- 説明の粒度調整
- 技術的修正と文体修正の分類

完成記事だけから推測した特徴は、このファイルの根拠にしない。

## ファイル間の責務

一つの現象が複数の階層に現れる場合は、各ファイルの抽象度に合わせて分解する。

例：

```text
thinking-flow.md:
読者が誤解する可能性を認識した場合、誤解が固定される前に補足する。

writing-style.md:
誤解が起きやすい説明の直後に、短い補足段落を配置する傾向がある。

stylistic-quirks.md:
補足の開始に「ここで注意したいのは〜です」を使用する場合がある。

refine-style.md:
AI草稿で離れた位置に置かれた注意事項を、関連する説明の直後へ移す傾向がある。
```

これは、同じ文を複数ファイルへ複製することを意味しない。

各ファイルでは、その責務に対応する情報だけを記述する。

配置に迷う場合は、次の問いで判断する。

- 「なぜそう書くか」という判断か
  - `thinking-flow.md`
- 「文章上、どこに何が置かれるか」という構造か
  - `writing-style.md`
- 「どのような言葉で書かれるか」という表現か
  - `stylistic-quirks.md`
- 「AI草稿をどう変更するか」という修正か
  - `refine-style.md`

## 共通の記述原則

### 観察結果ではなく、利用可能なルールにする

悪い例：

```text
具体例が多い。
```

良い例：

```text
抽象的な説明だけでは動作や用途を想像しにくい場合、概念説明の直後に具体例を置く傾向がある。
```

悪い例：

```text
文章がわかりやすい。
```

良い例：

```text
新しい専門用語を導入する前に、その用語が必要になる状況または既知の概念との関係を説明する。
```

### 条件を含める

ルールは、可能な限り次の形で記述する。

```text
〜の場合は、〜する。
ただし、〜の場合には適用しない。
```

常に適用する規則として表現するのは、十分な根拠がある場合に限る。

### 著者固有性を意識する

一般的な文章術として広く知られている原則を、そのまま著者固有の特徴として記載しない。

一般論と一致する場合でも、著者の文章でどのように現れているかを具体化する。

悪い例：

```text
読者にわかりやすく説明する。
```

良い例：

```text
定義から始めるよりも用途を先に示した方が理解しやすい概念では、必要性を示してから用語を導入する傾向がある。
```

### 主張の強さを根拠に合わせる

少数の記事でしか確認できない特徴に対して、「常に」「必ず」「一貫して」と書かない。

使用する表現の目安：

- 強い根拠がある
  - 「一貫して〜する」
  - 「多くの記事で〜する」
- 条件付きで確認できる
  - 「〜の場合に用いる傾向がある」
  - 「主に〜の記事で確認できる」
- 根拠が少ない
  - 「〜する場合がある」
  - 「現時点では弱い傾向として確認できる」

## ルールの基本形式

分析結果をルールとして記載する場合は、可能な限り次の形式を使用する。

```markdown
### ルール名

- 対象：
- ルール：
- 適用する状況：
- 目的：
- 適用しない状況：
- 確度：
- 根拠：
- 反例・例外：
- 関連ルール：
```

ファイル固有の項目が必要な場合は追加してよい。

### `thinking-flow.md`で追加できる項目

```markdown
- 認識する問題：
- 選択する説明戦略：
- 読者に起こそうとする理解の変化：
- 次の判断へ進む条件：
```

### `writing-style.md`で追加できる項目

```markdown
- 配置位置：
- 前後の要素：
- 記事タイプ：
- 構成上の効果：
```

### `stylistic-quirks.md`で追加できる項目

```markdown
- 表現：
- 使用目的：
- 使用されやすい文脈：
- 近いが著者らしくない表現：
- 過剰使用への注意：
```

### `refine-style.md`で追加できる項目

```markdown
- 修正前の特徴：
- 修正後の傾向：
- 修正操作：
- 修正の目的：
- 差分の種類：
```

すべての項目を機械的に埋める必要はない。意味のある情報がない項目は省略する。

ただし、少なくとも次は含める。

- ルール
- 適用条件
- 確度
- 根拠
- 適用しない条件または例外

## 確度の定義

確度は次の3段階を基本とする。

### 強い傾向

次の条件をおおむね満たすもの。

- 複数の記事で繰り返し確認できる
- 特定の一記事や一シリーズだけに偏っていない
- 異なる時期または異なる記事タイプでも確認できる
- 明確な反例が少ない
- 別の説明よりも、その特徴として解釈する方が妥当である

「強い傾向」は「例外なく適用する」という意味ではない。

### 条件付きの傾向

次のいずれかに該当するもの。

- 特定の記事タイプで繰り返し確認できる
- 特定の読者層や目的のときに確認できる
- 一定数の根拠はあるが、明確な反例もある
- 適用条件を限定すれば有効な特徴である
- 執筆時期によって変化している

条件を省略せずに記述する。

### 弱い傾向

次のいずれかに該当するもの。

- 確認例が少ない
- 根拠が一部の記事に偏っている
- 他の説明でも同じ現象を説明できる
- 反例が比較的多い
- 著者固有かどうか判断しづらい

弱い傾向は、生成時に積極的に適用するルールとして扱わない。

補助的な観察または今後の検証候補として記録する。

## 事実と推測の区別

各主張は、次のいずれかとして扱う。

### 直接確認できる事実

本文やGit差分から直接確認できるもの。

例：

- 定義の前に具体例が置かれている
- 「一方で」が複数の記事で使用されている
- AI草稿からまとめ段落が削除されている

### 強い推測

複数の観察結果から、比較的妥当に導けるもの。

例：

- 読者が用途を理解してから定義へ進めるようにしていると考えられる
- 断定を避けるために表現を調整している可能性が高い

### 限定的な推測

根拠はあるが、他の解釈も可能なもの。

例：

- 読者の混乱を避ける目的で段落を分けた可能性がある
- リズムを調整するために短文を置いた可能性がある

### 検証不足の仮説

興味深いが、現時点ではガイドへ採用できないもの。

検証不足の仮説は、主要ルールに混在させず、保留事項として分離する。

## 根拠の記載方法

根拠には、可能な限り次を含める。

```markdown
- 記事名：
- ファイル：
- セクション：
- 該当箇所の要約：
- 確認した現象：
```

Git差分の場合：

```markdown
- 記事名：
- 草稿コミット：
- 編集コミット：
- 対象ファイル：
- 差分の要約：
- 修正の分類：
```

### 根拠の数

単純な出現回数だけで確度を決めない。

次も考慮する。

- 記事数
- 記事タイプの幅
- 執筆時期の幅
- 同一シリーズへの偏り
- 同じ編集者や媒体による影響
- 反例の数
- 一つの特徴が文章全体へ及ぼす影響

### 根拠の引用

記事本文を長く転載しない。

用例は、特徴を確認できる最小限の長さにする。

全文や長い段落の複製ではなく、必要に応じて要約を使用する。

## 記事タイプと適用条件

記事タイプによって変化する特徴は、必ず条件を明記する。

記事タイプの例：

- 概念解説
- 実装解説
- チュートリアル
- 比較記事
- 導入記事
- エッセイまたは体験記事
- 短い告知文
- レビュー記事
- その他

例：

```markdown
- 適用する状況：
  概念解説で、読者が用語そのものをまだ知らない場合。

- 適用しない状況：
  APIリファレンスのように、読者が用語を既に理解していることを前提とする文章。
```

記事タイプだけでなく、次の条件も考慮する。

- 想定読者
- 記事の目的
- 記事の長さ
- 媒体の制約
- シリーズ内での位置
- 前の記事で説明済みの内容
- 技術的な厳密さが必要な度合い
- コードや図の有無

## 反例と例外の扱い

反例があることだけを理由に、ルールを直ちに棄却しない。

まず次を確認する。

- 記事タイプが異なるか
- 読者層が異なるか
- 記事の目的が異なるか
- 執筆時期が異なるか
- 媒体や編集者の影響があるか
- AI草稿由来の表現ではないか
- テーマ固有の制約があるか

反例によって適用条件を限定できる場合は、条件付きルールへ修正する。

例：

```text
修正前：
導入では具体例から始める。

修正後：
読者が概念の用途を想像しにくい場合、導入では定義より先に具体的な状況を示す傾向がある。
```

反例を説明できない場合は、確度を下げるか、保留にする。

## 用例の扱い

用例は、ルールを理解するための補助として使用する。

用例だけを並べてルールの説明を省略しない。

### 良い用例

- 短い
- ルールとの対応が明確
- 文脈が必要な場合は要約が付いている
- 特定テーマの専門用語に依存しすぎていない
- 著者の文章を大量に複製していない

### 避ける用例

- 記事の段落全体
- ルールと関係のない長い引用
- 一度しか使われていない特殊な表現
- 記事テーマ固有の専門用語
- 出典のない創作例を実例として扱うこと

創作した例を用いる場合は、「再現例」などと明示し、実際の記事からの用例と区別する。

## 重複と参照関係

同じ意味のルールを複数箇所へ記載しない。

既存ルールの一部として説明できる場合は、独立したルールを追加せず、次のいずれかで処理する。

- 既存ルールの条件を追加する
- 既存ルールの具体例として追加する
- 関連ルールとして参照する
- 上位ルールへ統合する
- 下位の表現例として配置する

関連する別ファイルのルールを参照する場合は、内容を全文コピーせず、次のように記述する。

```markdown
- 関連ルール：
  `thinking-flow.md`の「読者の誤解を先回りする判断」を参照。
```

見出し名は、参照先で安定して使用できる名前にする。

## 一般化の基準

次を満たさない特徴は、強いルールとして一般化しない。

- 複数の独立した記事で確認できる
- 特定テーマの必然性だけでは説明できない
- 一般的な文章術だけでは説明しきれない
- 明確な反例が少ない、または条件によって説明できる
- 記事生成や推敲時に実行可能な形へ変換できる

次は一般化しない。

- 一度だけ現れた言い回し
- 一記事だけの見出し構成
- テーマ固有の専門用語
- 単なる誤字修正
- 技術的正確性の訂正
- 媒体によって強制された構成
- 編集者の指示による可能性が高い変更
- AI草稿に残っていただけの表現
- 根拠のない執筆意図

## 既存ファイルの更新規則

既存ガイドがある場合は、原則として全面的に書き直さない。

新しい分析結果を、次のいずれかへ分類する。

- 補強
- 条件追加
- 例外追加
- 確度上昇
- 確度低下
- 新規ルール
- 既存ルールへの統合
- 別ファイルへの移動
- 廃止候補
- 保留

更新時は、次の形式で変更内容を記録する。

```markdown
### 更新対象

- 対象ファイル：
- 対象ルール：
- 更新区分：
- 更新前：
- 更新後：
- 更新理由：
- 新しい根拠：
- 反例：
```

この更新記録を最終ガイドへ常に残す必要はない。

分析ログまたは作業記録として保持し、最終ガイドには更新後のルールを反映する。

### 新しい記事が少数の場合

新しく追加された記事が一つまたは少数の場合は、既存の強いルールを即座に変更しない。

次の順序で検討する。

1. 既存ルールの例外か
2. 記事タイプによる違いか
3. 読者や目的による違いか
4. 執筆時期による変化か
5. 既存ルールの確度を下げる必要があるか
6. 新しいルール候補として保留すべきか

## 廃止と保留の扱い

### 廃止する場合

既存ルールを削除または廃止するのは、次のいずれかの場合とする。

- 根拠の誤読が判明した
- AI草稿由来の特徴だった
- 一般的な文章術以上の特徴ではなかった
- 反例が多く、条件による説明もできない
- 新しい分析により、別の上位ルールへ統合された
- 現在の著者の文章には当てはまらなくなった

削除前に、削除理由を記録する。

### 保留する場合

次の特徴は保留する。

- 根拠記事が少ない
- 別の解釈が有力である
- 記事タイプの偏りが大きい
- 反例の理由を説明できない
- 著者固有か判断できない
- 担当Agent間で結論が出ていない

保留した特徴を、生成時の主要ルールとして使用しない。

## Markdownの記述規則

### 見出し

- ファイル内では見出し階層を一貫させる
- `#`はファイルタイトルに一度だけ使用する
- 主要分類に`##`を使用する
- 個別ルールに`###`を使用する
- ルールの中で過度に深い見出しを使わない

### 箇条書き

箇条書きは、条件、根拠、例外などの構造化に使用する。

説明そのものを短い箇条書きの羅列だけで済ませない。

必要な箇所では文章で理由や関係を説明する。

### 表

表は、複数項目を同じ軸で比較するときに限って使用する。

長文を表のセルへ詰め込まない。

条件や例外を詳しく説明する必要がある場合は、見出しと箇条書きを使用する。

### コードブロック

コードブロックは、次の用途に限定する。

- 思考フロー
- 擬似コード
- 出力形式の例
- 修正前後の短い比較
- 再利用可能なテンプレート

通常の説明文をコードブロックへ入れない。

### 用語

次の用語は意味を統一する。

- ルール
  - 文章生成や推敲時に利用できる条件付きの指針
- 特徴
  - 記事から観察された現象
- 根拠
  - 特徴またはルールを支える本文やGit差分
- 反例
  - ルールに該当しない、または反対の現象が確認できる事例
- 例外
  - 条件によって説明可能な非適用事例
- 確度
  - 分析結果を著者固有の傾向として扱える強さ
- 推測
  - 本文に明示されていない意図や判断についての仮説

## ファイルごとの必須要素

### `thinking-flow.md`

最低限、次を含める。

- ガイドの目的
- 執筆前の判断
- 説明を進める基本的な思考の流れ
- 読者の疑問を扱う判断
- 抽象と具体を切り替える判断
- 説明を深める、または省略する判断
- 話題を展開する判断
- 推敲時の判断
- 再現可能な条件付きルール
- 適用上の注意
- 一般化できない判断

思考フローは、見出し順の言い換えではなく、問題認識と判断の連鎖として記述する。

### `writing-style.md`

最低限、次を含める。

- ガイドの目的
- 全体構成
- 導入
- セクション構成
- 段落構成
- 文の構成
- 定義と例の配置
- 補足や注意事項
- 記事タイプによる変化
- 生成時のガイドライン
- 避ける構成

### `stylistic-quirks.md`

最低限、次を含める。

- ガイドの目的
- 全体的な語り口
- 導入表現
- 説明を展開する表現
- 話題をつなぐ表現
- 例示表現
- 対比、否定、限定の表現
- 推測と断定の調整
- 文末表現
- 表記上の癖
- あまり使わない表現
- 再現時の注意

頻出表現の一覧だけで終えず、使用文脈と目的を記述する。

### `refine-style.md`

最低限、次を含める。

- ガイドの目的
- 修正時の基本方針
- 追加する傾向
- 削除する傾向
- 並べ替える傾向
- 書き換える傾向
- 構成を変更する傾向
- 技術説明を修正する傾向
- AIらしい文章を除去する基準
- 条件付きの修正
- 修正手順
- 一般化しない修正

修正傾向は、Git差分によって確認できるものを根拠とする。

## 最終検証

成果物を保存する前に、次を確認する。

### 責務

- 各ルールが適切なファイルに配置されているか
- 思考、構成、表現、修正傾向が混在していないか
- 同じ内容が複数ファイルへそのまま複製されていないか

### 根拠

- 各主要ルールに根拠があるか
- 根拠が主張を実際に支えているか
- 一つの記事だけを根拠に強い傾向としていないか
- Git差分を使う場合、修正前後の対応が明確か

### 確度

- 根拠の量と確度が対応しているか
- 条件付きの傾向から条件が抜け落ちていないか
- 弱い傾向が主要ルールとして書かれていないか

### 一般化

- 一般的な文章術を著者固有として扱っていないか
- 記事テーマ固有の特徴を一般化していないか
- 媒体や編集者の影響を無視していないか
- AI草稿由来の表現を著者の癖として扱っていないか

### 実用性

- 文章生成時に何をすればよいか判断できるか
- 適用条件と非適用条件がわかるか
- 固定テンプレートではなく判断ルールになっているか
- 頻出表現を機械的に挿入する指示になっていないか
- スタイルの物真似ではなく、自然な文章を作るためのガイドになっているか

### 更新可能性

- 新しい記事を追加した際に、既存ルールを更新できる形か
- 根拠と反例を追跡できるか
- ルールの統合、条件追加、廃止が可能な構造か

## 禁止事項

以下を行わない。

- 分析結果を記事の要約として出力する
- 観察された特徴を、条件のない命令へ変換する
- 根拠の少ない特徴へ強い確度を付ける
- 一つの記事や一つのシリーズだけから著者全体を一般化する
- 一般的な文章術を著者固有の特徴として記述する
- 記事テーマ固有の専門用語を表現上の癖として扱う
- AI草稿由来の表現を著者本人の特徴として扱う
- 技術的な誤りの訂正を文体修正として扱う
- 推測を事実として記述する
- 反例や例外を削除して、ルールを強く見せる
- 同じルールを複数ファイルへそのまま複製する
- 頻出表現をすべての文章へ使用するよう指示する
- すべての記事を同じ構成へ当てはめる
- 用例を大量に転載する
- 分析ログやAgent間の議論を、そのまま最終成果物へ含める
- 既存ガイドを理由なく全面的に書き直す
- 保留中の仮説を生成時の主要ルールとして使用する
