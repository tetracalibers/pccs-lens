# Output Contract（分析基準）

この文書は、著者スタイル分析の**分析・反証段階**で用いる判断基準の正典を定義する。

- `analysis-workflow.js` の分析／反証エージェントは、この文書を要約してスクリプト内へインライン展開した品質基準（`ANALYSIS_CRITERIA`）に従うため、通常この全文は Read しない。この文書はその基準の出所であり、保守・参照用に置く。
- 統合（Markdown 生成）時の記述形式・出力ルール・ファイル別必須要素は [`output-contract.md`](./output-contract.md) を参照する。
- 根拠・用例・著者固有性の詳細は分析と統合の双方に関わるため `output-contract.md` 側に残している。

## 確度の定義

確度は次の3段階を基本とする。

### 強い傾向

次の条件をおおむね満たすもの。

- 複数の記事で繰り返し確認できる
- 特定の一記事や一シリーズだけに偏っていない
- 異なる時期または異なる記事タイプでも確認できる
- 明確な反例が少ない
- 別の説明よりも、その特徴として解釈する方が妥当である

「強い傾向」は「例外なく適用する」という意味ではない。

### 条件付きの傾向

次のいずれかに該当するもの。

- 特定の記事タイプで繰り返し確認できる
- 特定の読者層や目的のときに確認できる
- 一定数の根拠はあるが、明確な反例もある
- 適用条件を限定すれば有効な特徴である
- 執筆時期によって変化している

条件を省略せずに記述する。

### 弱い傾向

次のいずれかに該当するもの。

- 確認例が少ない
- 根拠が一部の記事に偏っている
- 他の説明でも同じ現象を説明できる
- 反例が比較的多い
- 著者固有かどうか判断しづらい

弱い傾向は、生成時に積極的に適用するルールとして扱わない。

補助的な観察または今後の検証候補として記録する。

## 事実と推測の区別

各主張は、次のいずれかとして扱う。

### 直接確認できる事実

本文やGit差分から直接確認できるもの。

例：

- 定義の前に具体例が置かれている
- 「一方で」が複数の記事で使用されている
- AI草稿からまとめ段落が削除されている

### 強い推測

複数の観察結果から、比較的妥当に導けるもの。

例：

- 読者が用途を理解してから定義へ進めるようにしていると考えられる
- 断定を避けるために表現を調整している可能性が高い

### 限定的な推測

根拠はあるが、他の解釈も可能なもの。

例：

- 読者の混乱を避ける目的で段落を分けた可能性がある
- リズムを調整するために短文を置いた可能性がある

### 検証不足の仮説

興味深いが、現時点ではガイドへ採用できないもの。

検証不足の仮説は、主要ルールに混在させず、保留として本体ガイドから分離し、対応する保留プール `writing-guides/pending/<観点>.md` へ書く。

## 記事タイプと適用条件

記事タイプによって変化する特徴は、必ず条件を明記する。

記事タイプの例：

- 概念解説
- 実装解説
- チュートリアル
- 比較記事
- 導入記事
- エッセイまたは体験記事
- 短い告知文
- レビュー記事
- その他

例：

```markdown
- 適用する状況：
  概念解説で、読者が用語そのものをまだ知らない場合。

- 適用しない状況：
  APIリファレンスのように、読者が用語を既に理解していることを前提とする文章。
```

記事タイプだけでなく、次の条件も考慮する。

- 想定読者
- 記事の目的
- 記事の長さ
- 媒体の制約
- シリーズ内での位置
- 前の記事で説明済みの内容
- 技術的な厳密さが必要な度合い
- コードや図の有無

## 反例と例外の扱い

反例があることだけを理由に、ルールを直ちに棄却しない。

まず次を確認する。

- 記事タイプが異なるか
- 読者層が異なるか
- 記事の目的が異なるか
- 執筆時期が異なるか
- 媒体や編集者の影響があるか
- AI草稿由来の表現ではないか
- テーマ固有の制約があるか

反例によって適用条件を限定できる場合は、条件付きルールへ修正する。

例：

```text
修正前：
導入では具体例から始める。

修正後：
読者が概念の用途を想像しにくい場合、導入では定義より先に具体的な状況を示す傾向がある。
```

反例を説明できない場合は、確度を下げるか、保留にする。

## 一般化の基準

次を満たさない特徴は、強いルールとして一般化しない。

- 複数の独立した記事で確認できる
- 特定テーマの必然性だけでは説明できない
- 一般的な文章術だけでは説明しきれない
- 明確な反例が少ない、または条件によって説明できる
- 記事生成や推敲時に実行可能な形へ変換できる

次は一般化しない。

- 一度だけ現れた言い回し
- 一記事だけの見出し構成
- テーマ固有の専門用語
- 単なる誤字修正
- 技術的正確性の訂正
- 媒体によって強制された構成
- 編集者の指示による可能性が高い変更
- AI草稿に残っていただけの表現
- 根拠のない執筆意図
