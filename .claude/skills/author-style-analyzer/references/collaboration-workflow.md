# Collaboration Workflow

複数のAgentが、著者の文章と思考プロセスを分析し、相互レビューを通じてスタイルガイドを作成または更新する。

このワークフローの目的は、Agent同士で単に意見を交換することではない。それぞれが異なる分析観点から独立した仮説を立て、互いの根拠、一般化、分類、適用条件を検証することで、単独分析による偏りを減らすことである。

## 実行方式（Workflow）

このワークフローは、**Workflow ツールによる決定論的オーケストレーション**で実行する。正典スクリプトは `references/analysis-workflow.js`。teammate（Agent Teams / `SendMessage`）は使わない。teammate 同士のピア通信は「A が B のメッセージを待ち、B が A を待つ」形のアイドル・デッドロックでハングしやすく、進行役が LLM だとフェーズ受け渡しのポーリング漏れでも停止するためである。

決定論オーケストレーションでは次の性質が保証される。

- 制御フローが JS のため、**アイドル待ちが原理的に存在せず、ハングしない**。
- 各エージェントは走って**構造化出力（スキーマ準拠 JSON）を返して終了**する。idle プロセスが残らないので `TaskStop` による明示停止は不要。
- ステージ間の barrier（全結果が揃うまで待つ点）が JS で明示され、進行漏れが起きない。

teammate 方式の「Agent 同士が直接議論して反証する」性質は、**独立エージェントが他エージェントの構造化出力を反証検証する**形に置き換える。自由討論は失うが、反証の実効はスキーマ＋反証エージェントで担保する。

### 実行手順

1. **前処理（メインセッション）**：スコープを解決し、対象記事の絶対パス・記事タイプ・除外記事・Git 履歴を使える記事（草稿／編集コミット）・既存ガイド本体の出力先パス・保留プール `pending/` のパス（省略可、省略時はスクリプトが導出）・参照プロンプトのパスを整理して、`analysis-workflow.js` 冒頭コメントが定義する **manifest オブジェクト**を組み立てる。
2. **Workflow 実行**：`Workflow({ scriptPath: '.claude/skills/author-style-analyzer/references/analysis-workflow.js', args: <manifest> })` を呼ぶ。スクリプトが3ステージ（独立分析 → 反証・境界レビュー → 統合）を回す。
3. **後処理（メインセッション）**：戻り値（更新したガイド本体・保留プール・変更点・昇格した保留・保留事項）を検証し、ユーザーへ報告する。

### 成果物の受け渡し

最終成果物は、統合ステージのエージェントが **`writing-guides/` 配下のファイルへ直接書き込む**。中間結果（独立分析の特徴、反証・境界の verdict）はエージェントの構造化戻り値としてステージ間を流れ、ファイルには書き込まない。メインセッションは Workflow の戻り値と実ファイルで結果を確認する。

### Workflow を使えない場合

次のいずれかで代替する。いずれも「独立分析が終わるまで他の結果を見ない」「ステージ順を守る」原則は維持する。

- `Agent` ツール（サブエージェント）で各ステージを順に実行し、結果をリードへ集約する。エージェント同士の直接の相互レビューは行えないため、リードが反証・境界レビューと統合を担い、最終ファイルもリードが書き込む。
- 単一エージェントが役割を切り替えて順に担当する。ステージ順を守り、各役割の独立性を保つ。

## 基本原則

### 独立分析を先に行う

各Agentは、他のAgentの結論を読む前に、自分の担当領域を独立して分析する。

最初から他の分析結果を共有すると、先に示された仮説へ引っ張られ、異なる観点からの発見が失われる可能性がある。

独立分析が完了するまでは、次を共有しない。

- 他Agentが抽出した特徴
- 他Agentの暫定的な結論
- 採用予定のルール
- 統合担当の期待する結論

分析対象の記事、対象範囲、共通の出力条件は共有してよい。

### 担当領域を混同しない

各Agentは、自分の担当外の特徴を発見しても、自分の成果物へ無理に含めない。

担当外の特徴は、次の形式で引き継ぎ候補として記録する。

```text
- 発見した特徴：
- 本来の担当候補：
- 根拠：
- 自分の担当外と判断した理由：
```

### 合意を目的にしない

レビューの目的は、全員が同じ意見になることではない。

意見が一致しない場合は、無理に一つへ統合せず、次のいずれかとして扱う。

- 条件付きで採用する
- 弱い傾向として残す
- 判断を保留する
- 記事タイプごとの差として分離する
- 根拠不足として棄却する

### 主張ではなく根拠を比較する

Agent同士の議論では、「どちらの説明がもっともらしいか」ではなく、次を比較する。

- 根拠となる記事数
- 根拠となる具体的な箇所
- 記事タイプをまたいで確認できるか
- 時期をまたいで確認できるか
- 反例が存在するか
- 他の説明でも同じ現象を説明できないか
- 一般的な文章術ではなく、著者固有の特徴といえるか

### 推測の深さと確度を分ける

深い推測であること自体を理由に排除しない。

ただし、本文から直接確認できる特徴と、リバースエンジニアリングによる仮説を区別する。

各主張には、次のいずれかを付ける。

- 直接確認できる事実
- 複数の根拠に基づく強い推測
- 妥当だが限定的な推測
- 検証不足の仮説

## Agentの役割

必要に応じて、次の役割を割り当てる。

### Coordinator

分析全体を管理する。

担当：

- 分析対象の整理
- 記事の分類
- 各Agentへの担当割り当て
- 作業順序の管理
- レビュー論点の整理
- 重複、矛盾、未解決事項の記録
- 最終成果物の出力確認

Coordinator自身は、各分析結果の内容を先回りして決定しない。

### Thinking Flow Analyst

完成した記事から、著者の執筆時の問題認識、判断、説明戦略、読者モデル、思考の遷移を復元する。

`references/thinking-flow-analysis.md`に従う。

分析対象：

- なぜその説明を置いたか
- 直前にどのような問題を認識したか
- なぜその順番や説明方法を選んだか
- 読者の理解をどう変化させようとしたか
- 次の説明へどう接続したか

担当外：

- 語尾や頻出表現
- 単なる見出し構成
- 人間による修正差分

### Writing Structure Analyst

記事全体、セクション、段落、文の構成上の傾向を分析する。

`references/writing-style-analysis.md`に従う。

分析対象：

- 情報の配置順
- 導入と本題の接続
- セクション分割
- 段落構成
- 定義、例、理由、注意点の配置
- 記事タイプによる構成変化

担当外：

- 執筆中の内的判断の推測
- 頻出フレーズの収集
- 修正履歴の分析

### Stylistic Quirks Analyst

語彙、言い回し、文末、接続表現、表記など、文やフレーズのレベルに現れる癖を分析する。

`references/stylistic-quirks-analysis.md`に従う。

分析対象：

- 頻出表現
- 表現の使用文脈
- 断定や推測の調整
- 読者との距離感
- 記号や表記の傾向
- あまり使わない表現

担当外：

- 記事全体の構成
- 思考プロセス
- Git差分による修正傾向

### Revision Diff Analyst

Git履歴からAI草稿と人間による編集を比較し、修正傾向を分析する。

`references/refine-style-analysis.md`に従う。

分析対象：

- 情報の追加と削除
- 順序変更
- 表現変更
- 段落や見出しの変更
- AIらしい文章の除去
- 技術的修正と文体修正の分離

担当外：

- 完成記事だけから推測した特徴
- 単なる単語頻度
- コミット履歴で確認できない意図の断定

### Evidence Reviewer

各分析の根拠の強さを検証する。

担当：

- 根拠記事数の確認
- 引用箇所と主張の対応確認
- 反例の探索
- 一般化の範囲確認
- 事実と推測の区別
- 記事タイプや時期による偏りの確認

新しいスタイルルールを積極的に提案するのではなく、既存の主張を検証する。

### Boundary Reviewer

4つの成果物間の責務と境界を検証する。

担当：

- 同じ特徴の重複検出
- 配置先の妥当性確認
- 思考、構成、表現、修正の混同検出
- 上位概念と下位表現の整理
- 因果関係と相関関係の混同検出

### Synthesis Editor

レビュー後の分析結果を最終成果物へ統合する。

担当：

- 採用された特徴の整理
- 条件と例外の明記
- 重複の除去
- ファイル間の参照関係の整理
- 出力形式への変換
- 既存ガイド本体と保留プール（`pending/`）の差分更新
- 保留の管理（保留プールへの追記、根拠が増えた保留の本体への昇格、不要になった保留の削除）

各観点の Synthesis Editor は、自分の観点の「ガイド本体＋保留プール」の2ファイルだけを編集する。主要ルールは本体、保留は `pending/` に置き、両者で同じ特徴を重複させない。レビューで棄却または保留された特徴を独断で本体へ採用しない（保留は `pending/` へ）。

## 作業フェーズ

`analysis-workflow.js` は次の3ステージを順に回す。前段の barrier（全結果が揃う点）を JS で明示するため、独立分析とレビューが同時に走ることはない。

```text
[前処理：メインセッションが manifest を組み立て、Workflow を起動]
↓
ステージ1：独立分析（4エージェント並行）
↓  ── barrier：4分析すべてが揃うまで待つ ──
ステージ2：反証・境界レビュー（軽量ティアで実行）
   ・Evidence 反証（各分析を独立に検証）
   ・Boundary 境界レビュー（4分析を横断／barrier 必須）
↓
ステージ3：統合（4エージェント並行。観点ごとに本体＋pending 保留ファイルの2本を担当し、更新時は差分 Edit。保留の追記・昇格・削除もここで反映）
↓
[後処理：メインセッションが成果物を検証し、ユーザーへ報告]
```

teammate 方式の旧9フェーズは、次のようにこの3ステージへ畳み込まれている。以降の「フェーズ1〜9」の記述は、各ステージのエージェントが**適用すべきレビュー観点・判断基準の詳細**として参照する（独立した実行フェーズではない）。

| 旧フェーズ | 対応するステージ |
| --- | --- |
| 1 分析対象の整理 | 前処理（manifest 構築） |
| 2 独立分析 | ステージ1 |
| 3 一次相互レビュー ／ 4 反証と境界レビュー | ステージ2（Evidence 反証・Boundary 境界） |
| 5 改訂 ／ 6 統合レビュー | ステージ3（各ファイル担当の統合エージェントが verdict を反映して取り込む） |
| 7 成果物の作成 | ステージ3（4エージェントが各1ファイルを並行して書き込む） |
| 8 最終監査 | ステージ3内の自己確認 ＋ 後処理でのメインセッション検証 |
| 9 回収と報告 | 後処理（`TaskStop` は不要） |

記事数やAgent数が少ない場合でも、独立分析とレビューを同時に行わない。

## フェーズ1：分析対象の整理

Coordinatorは、分析開始前に次を整理する。

- 対象記事一覧
- 記事の保存場所
- 記事タイプ
- 執筆時期
- 対象から除外する記事
- Git履歴を使用できる記事
- AI草稿の起点となるコミット
- 既存ガイドの有無
- 今回、新規作成か差分更新か

記事タイプは、可能な範囲で次のように分類する。

- 概念解説
- 実装解説
- チュートリアル
- 比較記事
- 導入記事
- エッセイまたは体験記事
- その他

分類に迷う場合は、複数カテゴリを付けてもよい。

Coordinatorは、各Agentが同じ分析対象を見ていることを確認する。ただし、`refine-style`の分析ではGit履歴を利用できる記事だけを対象にしてよい。

## フェーズ2：独立分析

各担当Agentは、対応する分析プロンプトに従って作業する。

独立分析では、次を出力する。

```markdown
# Independent Analysis

## 分析対象

## 抽出した特徴

## 特徴ごとの根拠

## 適用条件

## 反例または例外

## 推測の確度

## 担当外として引き継ぐ特徴

## 判断を保留した特徴
```

各特徴は、次の形式で記述する。

```markdown
### 特徴名

- 主張：
- 分類：
- 根拠記事：
- 根拠箇所：
- 反例：
- 適用する記事タイプ：
- 適用しない条件：
- 確度：
- 事実または推測：
```

この段階では、他Agentの分析結果を参照しない。

## フェーズ3：一次相互レビュー

独立分析が完了した後、各Agentは他Agentの結果をレビューする。

レビューは、担当領域の正しさを競うものではない。自分の専門観点から、他の分析に不足している条件や別解を提示する。

### Thinking Flow Analystが確認すること

- 構成上の特徴の背後に、説明上の判断が隠れていないか
- 表現上の癖が、読者への働きかけと混同されていないか
- 修正傾向から、執筆時の判断を過度に逆算していないか
- 結果として現れた構成を、思考そのものと扱っていないか

### Writing Structure Analystが確認すること

- 思考上の仮説を、記事構成として断定していないか
- 頻出表現の位置や役割に構造的な条件がないか
- 修正傾向が特定の構成パターンに依存していないか
- 一つの記事形式の構成を全記事へ一般化していないか

### Stylistic Quirks Analystが確認すること

- 一般的な接続表現を著者固有として扱っていないか
- 構成上の役割と、表現上の癖が混同されていないか
- 頻度だけでなく使用文脈が示されているか
- 修正後に残った表現と、意図的に採用した表現を混同していないか

### Revision Diff Analystが確認すること

- 完成記事の特徴が、実際にはAI草稿由来ではないか
- 人間が一貫して削除している特徴が推奨されていないか
- 一度だけの修正が一般化されていないか
- 技術的修正と文体修正が混同されていないか

レビューコメントは、次の形式で記述する。

```markdown
### 対象となる主張

- レビュー結果：
  - 支持
  - 条件付きで支持
  - 要修正
  - 反対
  - 判断保留
- 指摘：
- 根拠：
- 反例：
- 修正案：
- 配置先に関する提案：
```

単なる感想や、根拠のない賛否は記述しない。

## フェーズ4：反証と境界レビュー

一次レビュー後、Evidence ReviewerとBoundary Reviewerが横断的に検証する。

### Evidence Reviewerの手順

各主張について次を確認する。

1. 根拠となる記事が複数存在するか
2. 同一シリーズや同一時期に偏っていないか
3. 引用箇所が主張を実際に支えているか
4. 記事テーマ固有の事情ではないか
5. 反例となる記事が存在しないか
6. 別の説明でも同じ現象を説明できないか
7. 強い確度を付けるだけの証拠があるか

判定は次から選ぶ。

- 根拠十分
- 条件を限定すれば妥当
- 根拠不足
- 反例が多い
- 著者固有とは判断できない
- 追加調査が必要

### Boundary Reviewerの手順

各特徴を次の基準で配置する。

#### `thinking-flow.md`

次に該当するもの：

- 執筆時に認識したと考えられる問題
- 説明方法を選ぶ判断
- 読者の状態に対する推測
- 次の説明へ進む条件
- 抽象と具体を切り替える判断

#### `writing-style.md`

次に該当するもの：

- 記事、セクション、段落、文の構成
- 情報の配置順
- 見出しや段落の分割方法
- 定義、例、理由、補足の配置
- 記事タイプ別の構成傾向

#### `stylistic-quirks.md`

次に該当するもの：

- 語彙
- フレーズ
- 文末
- 接続表現
- 記号や表記
- 語り口や断定の調整

#### `refine-style.md`

次に該当するもの：

- AI草稿と人間編集の差分
- 追加、削除、並べ替え、書き換え
- 人間が繰り返し修正する傾向
- AIらしい文章を除去する基準

一つの特徴が複数領域に関係する場合は、上位ルールと具体的な現れ方に分ける。

例：

```text
thinking-flow.md:
読者が誤解しそうな地点で、先回りして補足する。

writing-style.md:
誤解が生じやすい説明の直後に、短い補足段落を置く。

stylistic-quirks.md:
補足の導入に「ここで注意したいのは〜です」を使う傾向がある。

refine-style.md:
AI草稿で説明から離れた位置に置かれた注意事項を、関連箇所の直後へ移動する。
```

これは重複ではなく、異なる抽象度で同じ傾向が現れている状態である。

ただし、各ファイルで同じ文を繰り返してはならない。

## フェーズ5：担当Agentによる改訂

各担当Agentは、レビュー結果を受けて自分の分析を改訂する。

各指摘について、次のいずれかで回答する。

```markdown
### 指摘内容

- 対応：
  - 採用
  - 一部採用
  - 不採用
  - 保留
- 理由：
- 修正内容：
- 根拠：
```

指摘を不採用にする場合も、理由と根拠を示す。

単にレビュー内容をすべて受け入れない。また、元の結論を守ることを目的にしない。

改訂後の分析では、次を明示する。

- 追加した条件
- 確度を変更した特徴
- 削除した特徴
- 配置先を変更した特徴
- 保留した特徴

## フェーズ6：統合レビュー

改訂後、CoordinatorとSynthesis Editorが、4つの分析結果を並べて確認する。

次を検証する。

- 4ファイル間で矛盾していないか
- 同じ特徴を異なる言葉で重複していないか
- 上位の判断と具体的な現れ方が対応しているか
- 強い傾向と弱い傾向が区別されているか
- 条件と例外が失われていないか
- 反例が無視されていないか
- 一般論が著者固有の特徴として残っていないか
- 分析結果が実行可能なガイドへ変換できるか

各候補を次の状態に分類する。

```text
- 採用
- 条件付きで採用
- 弱い傾向として採用
- 保留
- 棄却
- 別ファイルへ移動
- 上位ルールへ統合
- 下位の具体例として保持
```

統合レビューでは、新しい仮説を大量に追加しない。新しい重要な仮説が出た場合は、担当Agentへ戻して検証する。

## フェーズ7：成果物の作成

Synthesis Editorは、採用された分析結果を次の成果物へ整理する。

- `thinking-flow.md`
- `writing-style.md`
- `stylistic-quirks.md`
- `refine-style.md`

成果物は、分析ログの寄せ集めではなく、文章生成や推敲で利用できるガイドとして記述する。

各ルールには、可能な限り次を含める。

- 何をするか
- どのような状況で行うか
- なぜ行うか
- 適用しない条件
- 確度
- 根拠
- 必要に応じて短い例

最終成果物には、Agent同士の議論の全文を含めない。

ただし、保留事項、反例、適用条件など、ガイドの正確性に必要な情報は残す。

## フェーズ8：最終監査

最終成果物を作成した後、担当外だったAgentが監査する。

推奨する割り当て：

- `thinking-flow.md`
  - Writing Structure AnalystとEvidence Reviewerが監査
- `writing-style.md`
  - Thinking Flow AnalystとBoundary Reviewerが監査
- `stylistic-quirks.md`
  - Revision Diff AnalystとEvidence Reviewerが監査
- `refine-style.md`
  - Writing Structure AnalystとBoundary Reviewerが監査

監査では、次を確認する。

```markdown
## Final Audit

- 担当領域に合っているか：
- 根拠が十分か：
- 過度な一般化がないか：
- 条件と例外が明記されているか：
- 他ファイルと重複していないか：
- 実際の生成や推敲に利用できるか：
- 修正が必要な箇所：
- 最終判定：
  - 承認
  - 条件付きで承認
  - 要修正
```

`要修正`となった場合は、Synthesis Editorが修正し、指摘した監査担当が再確認する。

## フェーズ9：後処理（メインセッションによる検証と報告）

統合ステージのエージェントが成果物を `writing-guides/` 配下のファイルへ書き込む。Workflow が戻ったら、メインセッションは次を行って初めて作業を完了とする。

1. **Workflow の戻り値を確認する。** 戻り値（各ファイルの `action`・変更点・昇格した保留 `promoted`・保留プールの更新 `pendingFiles`・保留事項 `heldItems`）を受け取り、独立分析がすべて失敗した等のエラーが無いことを確かめる。空・想定外の結果なら、トランスクリプトの `journal.jsonl` で各エージェントの実際の戻り値を確認する。
2. **成果物の存在と更新を検証する。** ガイド本体 `writing-guides/thinking-flow.md`・`writing-style.md`・`stylistic-quirks.md`・`refine-style.md` の4ファイルが実際に作成または更新されていることをファイルシステム上で確認する。保留の増減があった観点では、対応する保留プール `writing-guides/pending/<同名>.md` も更新されていることを確認する。
   - 既存ガイドの差分更新だった場合は、有効な既存記述が保持され、変更が加筆・修正・調整として反映されていることを確認する（全面的な書き直しになっていないか）。
   - 保留は本体ではなく `pending/` に置かれ、昇格した項目は本体へ移って `pending/` から消えていること、反例・適用条件が必要な範囲で記録されていること、各保留プール先頭の「このファイルの位置づけ」注記が残っていることを確認する。
3. **ユーザーへ報告する。** 更新したガイド本体・保留プールのパス、主な変更（新規追加・確度変更・削除・保留への追記・保留からの昇格）、未解決の保留事項を要約してユーザーへ伝える。

teammate を使わないため、待機プロセスの `TaskStop` は不要。Workflow を使わず単一エージェント／サブエージェントで代替した場合は、リード自身が最終ファイルを書き込んだうえで、同じ検証と報告を行う。

## 議論の形式

Agent間のやり取りは、自由形式の会話ではなく、論点単位で行う。

一つの論点について、次の順序を守る。

### 1. 提案

```markdown
- 主張：
- 配置先：
- 適用条件：
- 根拠：
- 確度：
```

### 2. 批判

```markdown
- 問題点：
- 根拠不足の箇所：
- 反例：
- 別の解釈：
- 修正案：
```

### 3. 応答

```markdown
- 批判を採用するか：
- 理由：
- 追加の根拠：
- 修正後の主張：
```

### 4. 判定

```markdown
- 結論：
- 配置先：
- 確度：
- 適用条件：
- 保留事項：
```

一つのAgentが提案、判定、統合をすべて担当しない。

## 意見が対立した場合

意見が対立したときは、次の順序で解決する。

1. 根拠となる本文またはGit差分を再確認する
2. 主張の対象範囲を狭められないか検討する
3. 記事タイプによる違いとして分けられないか検討する
4. 時期による文体変化として分けられないか検討する
5. 上位ルールと具体的な現れ方に分解できないか検討する
6. 確度を下げて残せないか検討する
7. 解決しなければ保留する

多数決だけで採用しない。

CoordinatorまたはSynthesis Editorの好みで決定しない。

## 既存ガイドを更新する場合

既存の4ファイルがある場合は、分析結果を全面的に置き換えない。

各新規発見を次のいずれかに分類する。

- 既存ルールを補強する
- 根拠を追加する
- 適用条件を狭める
- 適用範囲を広げる
- 例外を追加する
- 確度を上げる
- 確度を下げる
- 新しいルールとして追加する
- 古いルールを廃止する
- 判断を保留する

更新前後について、次を記録する。

```markdown
### 更新対象

- 既存の記述：
- 新しい分析結果：
- 更新区分：
- 更新後の記述：
- 更新理由：
- 根拠：
- 反例：
```

新しい記事が一つ追加されただけの場合は、その一例だけを理由に強いルールを変更しない。

既存ルールと矛盾する場合は、まず次を検討する。

- 記事タイプの違い
- 対象読者の違い
- 執筆時期の違い
- 編集者や媒体の影響
- AI草稿由来の表現
- テーマ固有の制約

## 禁止事項

以下を行わない。

- 独立分析前に他Agentの結論を共有する
- 全Agentで同じ分析を繰り返す
- 自由討論だけで結論を作る
- 多数決だけで採用を決める
- Coordinatorが分析結果を先に決める
- 他Agentへの批判を、根拠なく否定する
- 全員の意見を無理に一つへまとめる
- 根拠の弱い特徴を議論の勢いで採用する
- 一般的な文章術を著者固有の特徴として採用する
- 記事テーマ固有の語彙を文体の癖として採用する
- 思考、構成、表現、修正傾向を同じファイルへ混在させる
- AI草稿由来の特徴を著者本人の癖として扱う
- 技術的な訂正を文体上の修正として扱う
- 単発の事例を強い傾向へ昇格させる
- 反例を隠す、または無視する
- 推測を事実として記述する
- 議論ログをそのまま最終成果物へ貼り付ける

## 完了条件

次をすべて満たした場合に完了とする。

- 各担当Agentが独立分析を完了している
- 各分析に根拠、確度、適用条件が付いている
- 一次相互レビューが行われている
- Evidence Reviewerが根拠を検証している
- Boundary Reviewerが4ファイル間の境界を確認している
- レビュー結果を受けて担当Agentが分析を改訂している
- 統合レビューで重複と矛盾が整理されている
- 4つの成果物が適切な責務で作成または更新されている
- 最終監査が完了している
- 保留事項と反例が必要な範囲で記録されている
- 最終成果物が文章生成や推敲に利用できる状態になっている
- リードが4つの成果物ファイルの作成・更新を検証し、ユーザーへ変更内容を報告している
