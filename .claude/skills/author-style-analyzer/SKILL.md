# Author Style Analyzer

過去記事と編集履歴を分析し、著者固有の執筆スタイルを再利用可能なガイドとして整理する。

## 成果物

次のファイルを、リポジトリ直下の `writing-guides/` 配下に作成または更新する。

- `writing-guides/thinking-flow.md`
- `writing-guides/writing-style.md`
- `writing-guides/stylistic-quirks.md`
- `writing-guides/refine-style.md`

`writing-guides/` が存在しない場合は作成する。既存ガイドが別の場所にある場合は、リポジトリ内を検索し、その場所を優先して更新する。

各ファイルの役割を混同しない。

- `thinking-flow.md`
  - 完成した文章からリバースエンジニアリングした、著者の判断と思考の流れ
- `writing-style.md`
  - 記事全体、セクション、段落、文の構成上の傾向
- `stylistic-quirks.md`
  - 頻出する言い回し、語尾、接続表現、表記などの表現上の癖
- `refine-style.md`
  - AI草稿を人間が修正した際の追加、削除、並べ替え、書き換えの傾向

同じ特徴を複数のファイルへ重複して記載しない。分類に迷う特徴は、各分析結果を比較した後で配置先を決定する。

## 参照ファイル

分析内容に応じて次のファイルを読む。

- 思考プロセスの分析：
  `references/thinking-flow-analysis.md`
- 文章構成の分析：
  `references/writing-style-analysis.md`
- 表現上の癖の分析：
  `references/stylistic-quirks-analysis.md`
- Git履歴による修正傾向の分析：
  `references/refine-style-analysis.md`
- Agent間の協調と相互レビュー：
  `references/collaboration-workflow.md`
- 出力形式と共通ルール：
  `references/output-contract.md`

必要な参照ファイルだけを読む。

## 基本ワークフロー

### 1. 分析対象を確認する

次を確認する。

- 既存のスタイルガイドの有無（あれば内容を読み、更新は上書きではなく差分で行う）
- Git履歴を利用できるか
- 記事タイトルとコミットメッセージの対応規則

#### 参照する過去記事の指定

分析対象とする過去記事は、引数で指定する（引数は任意）。引数は「`#id`」「カンマ区切りの slug」「なし」の3通り。

- **`#id` が渡された場合**（先頭が `#`）：`app/src/lib/content-pages` 配下の YAML から該当 id を探し、その id に含まれる実装済み記事を対象にする。
  - id は YAML 上のカテゴリ／セクションの識別子。色記事はカテゴリ単位（`color-theory.yaml` / `color-fields.yaml` の top-level の `id`）、CG記事はセクション単位（`cg/*.yaml` の `sections[].id`）。
  - 該当 id の配下から、`slug:` を持つリンク（`PageLink`＝実装済みページ）をすべて集める。`title:` だけのリンク（`DraftLink`＝未実装でページが無い）は対象外。
  - 集めた slug を `app/src/routes/{cg,color-theory,color-fields}/**/<slug>/+page.svx` へ解決したものが分析対象。
  - `#id` は **1つだけ**指定できる（slug と異なり複数指定は不可）。id が見つからない、または複数の YAML で重複して見つかる場合は、その旨を報告して確認する。
- **カンマ区切りの slug が渡された場合**：指定された slug に対応する記事のみを対象にする。
  - slug は記事のディレクトリ名。`app/src/routes/{cg,color-theory,color-fields}/**/<slug>/+page.svx` を再帰的に探して特定する（CG記事は `cg/<分野>/<slug>/` のように階層が深いことがある）。
  - 見つからない slug は報告し、残りの slug で続行する。
- **引数が渡されていない場合**：解説記事（`layout: guide-content`）の `.svx` をすべて参照する。`app/src/routes/concept/+page.svx`（`layout: concept`）など、記事以外のページは除外する。
  - この場合は対象記事が多く、Agent Teams での分析はトークン消費が非常に大きくなる。**分析を始める前に必ず、ユーザーへ「大量のトークンを消費する恐れがある」ことを伝え、このまま続行してよいか確認する**（`AskUserQuestion` などで）。承認が得られるまで分析に進まない。範囲を絞りたい場合は `#id` または slug 引数を使うよう案内する。

いずれの場合も、フロントマターが `draft: true` の記事は分析対象から除外する（`draft` フィールドが無い記事は非draft＝公開済みとして扱う）。記事本文以外のファイル（デモコンポーネント、レイアウト、YAML など）は分析対象に含めない。

### 2. 記事を分類する

記事を可能な限り次の種類に分ける。

- 概念解説
- 実装解説
- チュートリアル
- 比較記事
- 導入記事
- エッセイまたは体験記事
- その他

記事タイプによる違いを無視して、すべての特徴を一律に一般化しない。

### 3. 各観点を独立して分析する

各分析は、先に他の分析結果へ引っ張られないよう独立して行う。

このワークフローは Claude Code の Agent Teams（teammates）機能での実行を想定している。有効な場合は、各役割を teammate に割り当てる（役割一覧と実行方式の詳細は `references/collaboration-workflow.md`、有効化に必要な設定は `README.md` を参照）。

- Thinking Flow Analyst
- Writing Structure Analyst
- Stylistic Quirks Analyst
- Revision Diff Analyst

Agent Teams を使えない場合は、`Agent` ツール（サブエージェント）や単一エージェントの役割切り替えで代替する。いずれの場合も、独立分析が終わるまで他の結果を見ない。各担当は、自分の対象外の特徴を成果物へ混在させない。

### 4. Agent間で相互レビューする

独立分析後、各結果を相互レビューする。

確認項目：

- 単一記事の特徴を一般化していないか
- 一般的な技術記事の特徴を著者固有の癖としていないか
- 根拠の弱い推測を断定していないか
- 他の成果物と内容が重複していないか
- 記事タイプによる条件を落としていないか
- 反例となる記事が存在しないか

意見が分かれた特徴は、無理に採用しない。次のいずれかに分類する。

- 採用
- 条件付きで採用
- 保留
- 棄却

### 5. 根拠と確度を整理する

各特徴には、可能な限り次を付ける。

- 確度
  - 強い傾向
  - 条件付きの傾向
  - 弱い傾向
- 確認できた記事数
- 対象となる記事タイプ
- 反例の有無
- 事実か推測か
- 最終確認日

記事本文の長い転載は避ける。用例は特徴を確認できる最小限の長さにする。

### 6. 既存ガイドを差分更新する

既存ファイルがある場合は、**まずその内容を読み、有効な記述は保持する**。全面的に書き直して既存内容を破棄することはしない。新しい分析結果は、既存記述への加筆・修正・調整として反映し、次のいずれかに分類する。

- 既存ルールを補強する
- 適用条件を追加する
- 例外を追加する
- ルールの確度を下げる
- 新しいルールを追加する
- 根拠不足のルールを削除する

変更理由が追跡できる形で更新する。

## Git履歴の分析

`refine-style.md`を作成または更新するときは、Gitのコミット履歴を使用する。

コミットメッセージを次の規則で解釈する。

- `記事タイトル：草稿を書かせた`
  - AIによる初期草稿
- 同じ`記事タイトル：`で始まる、その後のコミット
  - 人間による編集

同一記事のコミットを時系列に追跡し、AI草稿から人間による編集までの変更を分析する。

次の変更は、文体上の修正から分離する。

- 誤字脱字のみの修正
- メタデータのみの変更
- リンク更新
- フォーマット整形
- 技術的誤りの訂正
- 記事内容とは無関係なリファクタリング

Git差分を文字単位で列挙するだけで終えず、修正の目的を分類する。

## 出力時の共通ルール

- 一般論ではなく、著者固有の特徴を記述する
- 固定テンプレートではなく、条件付きの判断として表現する
- 頻出するだけの表現を機械的に推奨しない
- 反例や適用しない条件を記載する
- 推測は推測であることを明示する
- 根拠のない心理状態を断定しない
- 著者の文章を過度に単純化しない
- 4つの成果物の責務を保つ

## 完了条件

次を満たした場合に完了とする。

- 4つの成果物が作成または更新されている
- 各特徴の配置先が適切である
- 強い傾向と弱い傾向が区別されている
- 記事タイプによる差が反映されている
- 相互レビューで指摘された重複と矛盾が解消されている
- 既存ガイドを更新した場合、変更内容が説明できる
