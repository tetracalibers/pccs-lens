# Design Doc Updater

`app/` 配下の実装から Color Prism のデザインシステムを抽出し、リポジトリ直下の `DESIGN.md` を実装に合わせて更新する。DESIGN.md は Figma などの外部仕様ではなく **実装（コード）を正としてリバースエンジニアリングした記述** である点が前提。

## 成果物

リポジトリ直下の `DESIGN.md` を作成または更新する。

- 存在しない場合は新規作成する。
- 別の場所にある場合はリポジトリ内を検索し、その場所を優先して更新する。
- 更新は **上書きではなく差分**で行う（後述の「差分更新の原則」）。

## 引数（任意）

引数で **分析範囲（スコープ）** を指定できる。**引数が指定された場合は、その範囲に限って分析し、DESIGN.md の該当箇所だけを加筆修正する**（範囲外のセクションは分析も編集もしない）。複数の引数をカンマ／空白区切りで併記してもよい。

引数は次の種類を受け付ける。どの種類かは「引数の解決」の手順で判定する。

- **なし**：全観点を分析し、DESIGN.md 全体を差分更新する。
- **観点／セクション名**（例：`カラー` / `タイポグラフィ` / `スペーシング` / `レイアウト` / `レスポンシブ` / `コンポーネント`）：その観点だけを再分析し、対応セクションのみ加筆修正する。
- **コンポーネント名**（例：`ToneSelector` / `FamilyCard` / `SiteHeader`）：該当する `.svelte` を特定し、そのコンポーネントのスタイルだけを分析する。観測したパターン（角丸・ボーダー・影・トランジション・フォーカス・ホバー・新規トークン等）を、対応するセクション（多くは「コンポーネント」。新トークンを導入していれば「カラー」等も）へ加筆修正する。
- **ディレクトリ名／パス**（例：`lib/components/jis-color-map` / `lib/demo/pccs` / `routes/patterns`）：配下のスタイルを持つファイルを横断分析し、観測されるパターン・値を反映する。`lib/demo/**` は図解＝プロース規約の例外である点を保つ。
- **ブランチ名**（例：`feat/lightness-match`）：そのブランチが加えた **デザイン関連の差分だけ** を分析して反映する（後述「ブランチスコープ」）。
- **変更ファイル／変更内容のヒント**（例：「color.css を変えた」「トグルの角丸を変えた」）：その変更が影響する観点だけに絞って分析・更新する。

範囲が曖昧なとき（引数がどの種類にも一意に解決できない、複数に該当する等）は、加筆修正の前に対象を確認する。

### 引数の解決

引数の種類は次の順で判定する。

1. **ブランチ／Git ref**：`git rev-parse --verify --quiet <arg>`、`git branch --list <arg>`、`git branch -r --list "*<arg>"` のいずれかで参照が見つかれば、ブランチ（または任意の ref）として扱う。
2. **ディレクトリ**：`<arg>` または `app/src/<arg>` 等がディレクトリとして存在すれば、ディレクトリスコープ。
3. **コンポーネント／ファイル**：`.svelte`（または `.css` / `.ts`）名として `app/src` 配下を再帰検索し一意に解決できれば、コンポーネントスコープ（複数該当時は確認）。
4. **観点キーワード**：「分析観点と対象ファイル」表の観点名に一致すれば、観点スコープ。
5. いずれにも解決できなければ、自由記述のヒントとして扱い、影響しそうな観点を推定する（曖昧なら確認）。

同名衝突（例：ブランチ名とディレクトリ名が同じ）で判定が割れる場合は、ユーザーに確認する。

### ブランチスコープ

ブランチ名が渡された場合は、そのブランチで加わった変更のうち **デザインに関わるものだけ** を反映する。

1. **基準を決める**：既定は `main`（`git merge-base main <branch>` を基点にする）。別基準を使う場合は引数で指定させる。
2. **差分を取る**：`git diff <base>...<branch> -- 'app/src/**'` から、スタイルを持つ変更（`.svelte` の `<style>` / `.css` / トークン定義 `.ts`）に絞る。
3. **変更ハンクだけを分析する**：追加・変更・削除されたトークン、値の変更、新しいコンポーネントやパターンを抽出する。`lib/demo/**` の図解の色変更などプロース規約の例外は反映しない（必要なら「デモ・図解の注意」に限って触れる）。
4. **反映する**：抽出したデルタを DESIGN.md の該当セクションへ加筆修正する。値が変わっていれば連動してチェックリストも見直す。
5. **更新対象の所在に注意**：分析したブランチと、更新する DESIGN.md が別のワークツリー／ブランチにあることがある。その場合は、現在の作業ツリー直下の DESIGN.md を更新し、**どのブランチの差分を反映したか** を明記する。

## 分析観点と対象ファイル

次の観点ごとに実装を調べ、対応するセクションへ反映する。値（トークン名・数値・セレクタ）は必ずソースで確認し、`file:line` を根拠として押さえる。**推測で書かない。**

| 観点 | 主な対象 | DESIGN.md の反映先セクション |
| :--- | :--- | :--- |
| カラー＆テーマ機構 | `app/src/lib/styles/color.css`（色トークン）、`app/src/routes/+layout.svelte`（`body.light/.dark`・`color-scheme`）、`app/src/lib/state/lightMode.svelte.ts`・`app/src/lib/components/SwitchLightDark.svelte`（切替機構） | カラー |
| タイポグラフィ | `app/src/routes/+layout.svelte` の `:global(body)`（`--font-*` 定義・`<svelte:head>` のウェブフォント）、`app/src/lib/components/Heading1.svelte`（ページ見出し）、`app/src/lib/components/m-html/Heading{2,3,4}.svelte`（本文見出し）、`app/src/lib/layouts/guide-content.svelte` | タイポグラフィ |
| スペーシング・レイアウト | `app/src/routes/+layout.svelte`（`.container`・ページシェル）、`app/src/lib/layouts/*`（読み幅・レイアウト種別）、`app/src/lib/components/site-layout/*` | スペーシング／レイアウト |
| レスポンシブ | `@media` と `container-type` / `cqw`・`cqh` の使用箇所を横断（`grep`） | レスポンシブ |
| コンポーネント・UIパターン | `app/src/lib/components/**`（`border-radius`・`border`・`box-shadow`・`transition`・フォーカス／`:focus-visible`・ボタン・ホバーの共通イディオムを横断） | コンポーネント |

補足：

- **ドメインの色見本データ（PCCS/JIS/Munsell）は `app/src/lib/data/*.json`** にあり、UI テーマ色（`color.css`）とは別物。DESIGN.md では所在の言及にとどめ、データ本体は転載しない。
- **デモ・図解（`app/src/lib/demo/**`）はプロース規約の例外**。色は図の一部としてハードコード HEX や局所 `light-dark()` を許容する。この区別を DESIGN.md でも保つ。

## 基本ワークフロー

### 1. 既存 DESIGN.md を読む

既存があれば **必ず内容を読み**、現在のセクション構成・記述・チェックリストを差分更新の起点にする。無ければ新規作成として進む。

### 2. 分析範囲を確定する

引数から分析範囲を確定する（「引数（任意）」「引数の解決」を参照）。

- **引数なし**：全観点を分析し、DESIGN.md 全体を差分更新する。
- **引数あり（スコープ指定）**：指定された範囲（観点／コンポーネント／ディレクトリ／ブランチ）に **限って** 分析し、DESIGN.md の該当箇所だけを加筆修正する。**範囲外のセクションは分析も編集もしない。**
  - 例外として、スコープ内で観測した事実が別セクションへ波及する場合（例：あるコンポーネントが新しい色トークンを導入 → 「カラー」にも追記、値変更 → 「チェックリスト」も整合）は、その波及先に限って触れてよい。波及させた箇所は報告する。

### 3. 各観点を独立して分析する

観点ごとに実装を調べ、**正確なトークン名・値・セレクタ・`file:line`** を収集する。観点が複数にわたる全体更新では、`Agent` ツールで観点ごとに独立したサブエージェントを並行起動し、各自が構造化した調査結果（トークン表・実測値・参照）を返す方式が効率的（本ドキュメントもこの方式で初版を作成）。単一観点の小さな更新はインラインで行ってよい。

各観点の調査では次を必ず確認する。

- **トークン化されているか / いないか**を偽らない。色・フォントファミリーは CSS 変数でトークン化されている一方、**サイズ・余白はトークン化されておらず `rem` 直値**で運用されている。存在しない `--space-*` スケール等をあると書かない。
- 相対色構文（`oklch(from …)` / `hsl(from …)` / `rgb(from …)` / `color-mix()`）や `light-dark()` の導出式は、式のまま正確に写す。
- 「実際に使われている非公式スケール」（頻出のサイズ帯・余白帯・`border-radius` 値・`transition` の二層など）は、分布として観測し記述する。

### 4. 実装との乖離を洗い出す

既存 DESIGN.md の記述と実装の実測値を突き合わせ、次を検出する。

- 値のズレ（トークン値・幅・ブレークポイント等が変わった）
- 追加・削除されたトークンやコンポーネントパターン
- 規約と実装の矛盾（例：DESIGN.md は角丸標準と書いているのに新規に `border-radius: 0` が出現、テキスト色に `--color-body` 直接参照の規約に反する局所 `light-dark()` 再定義が増えた 等）

矛盾は「記述を実装に合わせて更新する」のか「実装側の逸脱として指摘する」のかを区別して扱い、後者は報告する（DESIGN.md を勝手に緩めない）。

### 5. DESIGN.md を差分更新する

「差分更新の原則」に従って更新する。値を変えたら、それに連動する **末尾のチェックリストも整合させる**。

### 6. 検証・報告する

更新後、次を行う。

- 記載した主要な値がソースと一致することをファイル上で再確認する。
- ユーザーへ変更内容を報告する：更新パス、変更したセクションと要点、検出した実装との乖離（採用した更新／実装側の逸脱として保留した指摘）。

## 差分更新の原則

- 既存の有効な記述は **保持** する。全面書き直しで既存内容を破棄しない。
- 新しい分析結果は、既存記述への加筆・修正・値の更新・例外の追加として反映する。
- DESIGN.md の **セクション構成を保つ**（設計思想／基本原則／本ドキュメントの位置づけ／カラー／タイポグラフィ／スペーシング／レイアウト／レスポンシブ／コンポーネント／デモ・図解の注意／コード生成時のチェックリスト）。構成自体を変える場合は理由を添えて提案する。
- 記述の性質を歪めない：トークン化の有無、「コードが正」という位置づけ、プロースと図解の規約差を維持する。
- 値には根拠（`file:line` で確認した実測値）を持たせ、推測は書かない。
- **スコープ指定時は、指定範囲に根拠のある変更だけ** を行い、範囲外の記述は分析も編集もしない（意図しない全体書き換えを避ける）。波及先に触れる場合もその旨を報告する。

## 完了条件

- `DESIGN.md` が作成または更新されている。
- 記載した主要な値が現在のコードと一致している（`file:line` で確認済み）。
- セクション構成が保たれ、変更に連動してチェックリストが整合している。
- トークン化の有無・「コードが正」・プロース／図解の規約差が崩れていない。
- スコープ指定時は、指定範囲（＋報告済みの波及先）の外を変更していない。
- 変更内容（対象スコープ・更新セクション・要点・実装との乖離。ブランチスコープ時は反映元ブランチと基準）をユーザーへ報告している。
