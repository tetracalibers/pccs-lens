# PR説明文更新スキル

引数で受け取った GitHub PR について、**そのPRの変更内容に合わせてタイトルと説明文（本文）を更新する**ためのスキルです。

特に、`+page.svx`（記事ページ）の差分を調べ、**`draft: true` を外して公開された記事**を検出し、説明文にカテゴリ別でリストアップします。

## 入力

- **PRのURL**（必須）: `https://github.com/tetracalibers/pccs-lens/pull/<番号>` の形式。`gh` コマンドにはこの URL をそのまま渡せる（`gh pr view <URL>` / `gh pr diff <URL>` / `gh pr edit <URL>`）。

## 前提知識

### 記事の公開（draft）規約

記事ページ（`+page.svx`）の下書き・公開は frontmatter の `draft` 行で表現される。

- **下書き**: frontmatter に `draft: true` 行がある（`grades:` の下に空行を挟んで置かれる）。
- **公開**: `draft: true` 行（および直前の空行）を**削除**する。`draft` 行そのものが無くなる。

したがって「draft を外して公開された記事」とは、**そのPRの差分で `draft: true` 行が削除された `.svx` ファイル**を指す。

### `gh pr diff` の制約

- `gh pr diff <URL>` は**パス指定（`-- <path>`）を受け付けない**（`accepts at most 1 arg(s)` エラーになる）。ファイルごとに絞り込めないので、**全体の diff を1回取得し、`diff --git` セクション単位でパースする**こと。
- diff の各ファイルセクションには frontmatter の `title:` 行が文脈行（先頭スペース）または追加行（`+`）として含まれるので、そこから**日本語タイトルを抽出できる**。

### カテゴリ（ルート）と日本語見出しの対応

公開記事は `app/src/routes/<カテゴリ>/<slug>/+page.svx` に置かれる。説明文の見出しには次の日本語名を使う（`app/src/routes/+page.svelte` のトップページ定義が正典）。

| ルート（`<カテゴリ>`） | 日本語見出し |
| --- | --- |
| `color-theory` | 色の理論 |
| `color-fields` | 色の活用分野 |
| `cg` | 色と画像（CGと画像処理） |

表にないカテゴリが出た場合は、`app/src/routes/+page.svelte` の該当 `title` を確認して見出しを補う。

## 手順

### 1. PRのメタ情報を取得する

```
gh pr view <URL> --json number,title,body,url,baseRefName,headRefName,state
```

- `state` が `OPEN` でない（`MERGED` / `CLOSED`）場合は、その旨をユーザーに伝えて、続行してよいか確認する。
- 既存の `body`（説明文）に**人手で書かれた内容がある**場合は、勝手に上書きしない。内容を確認し、置き換えてよいか／末尾に追記すべきかをユーザーに確認する。空（または本スキルが生成したと分かる定型文）なら、そのまま新しい説明文で置き換えてよい。

### 2. 変更ファイルと全体の差分を取得する

```
gh pr diff <URL> --name-only          # 変更ファイル一覧
gh pr diff <URL>                       # 全体の unified diff（パースに使う）
```

### 3. 公開された記事（draft を外した記事）を検出する

全体の diff を `diff --git` セクション単位でパースし、`.svx` ファイルのうち **`draft: true` 行が削除され、かつ `draft: true` 行が再追加されていない**ものを「公開された記事」とする。同じセクションの `title:` 行からタイトルを、パスからカテゴリを取り出す。

次のワンライナーがそのまま使える（`<URL>` を置換）:

```sh
gh pr diff <URL> | awk '
  function flush() {
    if (file ~ /\.svx$/ && removedDraft && !readdedDraftTrue) {
      cat=file; sub(/^app\/src\/routes\//,"",cat); sub(/\/.*/,"",cat)
      printf "PUBLISHED\t%s\t%s\t%s\n", cat, title, file
    }
  }
  /^diff --git / {
    flush()
    file=$0; sub(/^diff --git a\/.* b\//,"",file)
    removedDraft=0; readdedDraftTrue=0; title=""
  }
  file ~ /\.svx$/ && $0 ~ /^[ +]title: /    { t=$0; sub(/^[ +]title: /,"",t); if(title=="") title=t }
  file ~ /\.svx$/ && $0 ~ /^-draft: *true/  { removedDraft=1 }
  file ~ /\.svx$/ && $0 ~ /^\+draft: *true/ { readdedDraftTrue=1 }
  END { flush() }
'
```

出力は `PUBLISHED<TAB>カテゴリ<TAB>タイトル<TAB>パス` の形。

**エッジケース:**

- **公開された記事が1件も無い場合**: 「公開した記事」節は作らず、手順 4〜6 で変更内容の要約だけで説明文とタイトルを整える。
- **新規追加（status `A`）で最初から `draft: true` が無い記事**: 「draft を外して」公開したわけではないため、原則リストには含めない。ただし実質的に新規公開ページなので、その存在をユーザーに伝え、「公開した記事」に含めるべきか確認する（このリポジトリでは新規ページは `draft: true` で作るのが通常なので稀）。

### 4. その他の主な変更を要約する

説明文の「その他の主な変更」節と、新しいタイトルを決めるために、PR全体の変更を把握する。

```
gh pr view <URL> --json commits --jq '.commits[].messageHeadline'
```

- コミット件名と変更ファイル一覧（手順 2）から、公開記事以外の主な変更を日本語で数点にまとめる（例: 新規ページ雛形・下書きの追加、既存ページの加筆、ヘッダーナビの調整、ドキュメント・スキル・スタイルガイドの更新など）。
- 細かなコミット（`[skip ci]` の規約追記や軽微な修正）は個別に列挙せず、まとめて要約する。

### 5. 説明文（本文）を組み立てる

下記の雛形（PR #87 の説明文がベース）に沿って Markdown を組み立てる。公開記事はカテゴリごとに `###` でまとめ、`- **タイトル** — パス` の形で列挙する。

```markdown
## 公開した記事（draft を外して公開）

このブランチで下書き（`draft: true`）を外して公開した記事は次の <件数> 件です。

### <日本語カテゴリ名（例: 色の理論）>
- **<記事タイトル>** — `app/src/routes/<カテゴリ>/<slug>/`
- ...

### <日本語カテゴリ名（例: 色の活用分野）>
- **<記事タイトル>** — `app/src/routes/<カテゴリ>/<slug>/`
- ...

## その他の主な変更

- <変更点1>
- <変更点2>
- ...
```

- パスは `+page.svx` を除いたディレクトリまで（`.../<slug>/`）を `code` で示す。
- 公開記事が無い場合は「公開した記事」節を省き、「その他の主な変更」（または「変更内容」）節のみとする。
- カテゴリの並び順・記事の並び順は、`app/src/routes/+page.svelte` のカテゴリ順や自然な流れに合わせて読みやすく整える。

### 6. タイトルを決める

そのPRの変更内容全体を端的に表す日本語タイトルにする。公開記事がある場合はそれが主眼になることが多い。

- 例: `色の理論・活用分野の記事を追加し、4記事を公開`
- コミット件名やブランチ名を機械的に流用せず、**PR全体の成果**が伝わる要約にする。

### 7. PRのタイトルと説明文を反映する

説明文は改行・Markdown を保つため、一時ファイル（スクラッチパッド等）に書き出してから `--body-file` で渡す。

```
gh pr edit <URL> --title "<新しいタイトル>" --body-file <一時ファイルのパス>
```

反映後、`gh pr view <URL> --json title,url --jq '.title, .url'` で結果を確認する。

### 8. ユーザーに報告する

- 検出した「公開された記事」の一覧（タイトル・カテゴリ・パス）。
- 変更前後の PR タイトル。
- 説明文に記載した「その他の主な変更」の要点。
- タイトルは要約であり、意図に合わせて調整可能である旨を添える。

## 注意事項

- **勝手に既存の説明文を破棄しない。** 人手で書かれた本文がある場合は、置き換え/追記の方針をユーザーに確認する（手順 1）。
- 公開判定は必ず**svx の差分の `draft: true` 行削除**で行う。YAML（`content-pages/*.yaml`）の `DraftLink`→`PageLink` 置換だけでは、記事本文の公開状態を正しく判定できないので根拠にしない。
- `gh pr diff` はパス指定不可。全体 diff をパースすること（手順 2・3）。
- URL は `gh` の各サブコマンドにそのまま渡せる。フォーク由来の PR ではヘッドが別リポジトリになる点に注意（`gh pr diff/view/edit` は URL 指定なら問題なく動く）。
- 本スキルはコード変更を伴わない（PR のメタ情報更新のみ）ため、`npm run check` は不要。
