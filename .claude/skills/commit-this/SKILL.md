# コミット作成スキル (commit-this)

**現在のセッションで変更したファイルだけ**を対象に、CLAUDE.md の「Git 規約」「品質チェック」に沿ったコミットを 1 つ（必要なら複数）作るためのスキルです。

このスキルがやること・やらないことを最初に確認する。

- **やること**: セッションで触ったファイルの特定 → コミット対象の分類 → 未 push コミットを見た `[skip ci]` 判断 → 日本語コミットメッセージの組み立て → 明示パス指定でのコミット。
- **やらないこと**: `push` はしない。`git add -A` / `git add .` / `git commit -a` は使わない。**セッション外（ユーザーや別セッションが加えた）未コミット変更を巻き込まない。**

## 入力

- **補足（任意）**: コミットメッセージのヒント、対象を絞る指示（「◯◯だけ」）、依頼の意図など。無くても動く。

## 手順

### 1. 現在のセッションで変更したファイルを特定する

コミット対象は、**このセッションであなた（アシスタント）が作成・編集したファイル**に限る。

1. **セッションでの変更を思い出す。** このセッション中に Write / Edit / NotebookEdit で作成・編集したファイル、および Bash で動かした（`mv`・リネーム・フォーマッタ・生成物など）ファイルを列挙し、これを「セッション変更リスト」とする。これが対象判定の一次情報。
2. **`git status` と突き合わせる。**

   ```sh
   git status --porcelain
   ```

   - セッション変更リストのうち、実際に差分がある（`M` / `A` / `??` / ステージ済み）ものだけを残す。差分が無いものは編集後に元へ戻った等なので除外する。
   - **git 上は dirty だが、セッション変更リストに無いファイル**は、セッション開始時点で既に存在した変更（ユーザーや別セッション由来）とみなし、**コミットに含めない**。存在は手順 6 で報告する。
3. **巻き込みのエッジケースを確認する。** セッションで触ったファイルが、セッション開始時点で**既に別の未コミット変更を持っていた**場合（環境のセッション開始時 `git status` スナップショットに載っていたファイルなど）、そのファイルをステージするとセッション外の差分も一緒にコミットされる。該当があれば、その旨をユーザーに伝えて含めてよいか確認する。
4. **セッションの記録が無いとき**（別セッションで加えた変更を後からコミットする等で、自分が編集した記憶が無い場合）は推測しない。`git status` を提示し、どのファイルを対象にするかユーザーに確認する。

> `git add -A` / `git add .` / `git commit -a` は使わない。必ず**パスを明示**してステージする（手順 5）。

### 2. コミット対象を分類する（app・ビルド系 / ドキュメント・スキル系）

`[skip ci]` の判断（手順 3）とメッセージ規約（手順 4）のため、対象ファイルを次の 2 種に分類する。

- **app・ビルド系（CI 対象）**: `app/**`（`.svx`・`.css` を含む）、ルートの `package.json`・lockfile・`*.config.*`・CI 設定など、**アプリの実行・ビルドに関わる**ファイル。
- **ドキュメント・スキル系（CI 非対象）**: `docs/**`、`spec/**`、`.claude/skills/**`、ルートの `*.md`（`CLAUDE.md` / `README` など）、その他アプリの実行に関わらない Markdown。

> **注意**: `.svx` は `app/` 配下の記事ページでビルド対象なので、**app・ビルド系**として扱う（`[skip ci]` の対象外）。「Markdown だから CI 非対象」と早合点しない。

### 3. `[skip ci]` を付けるか判断する

CLAUDE.md の規約をそのまま適用する。判断材料として**未 push コミット**を確認する。

```sh
# 未 push コミット一覧（upstream 未設定でも動く：リモートに存在しないコミットを列挙）
git log --oneline HEAD --not --remotes

# 未 push コミットが触れているファイル一覧
git log HEAD --not --remotes --name-only --pretty=format: | sort -u | sed '/^$/d'
```

判断ルール:

- `[skip ci]` を付けてよいのは、**今回のコミット対象が丸ごとドキュメント・スキル系のみ**（app・ビルド系を 1 つも含まない）の場合だけ。
- **さらに、既存の未 push コミットに app・ビルド系の変更が 1 つでも含まれる場合は、たとえ今回のコミットがドキュメント・スキルのみでも `[skip ci]` を付けない。** 先頭コミットの `[skip ci]` は push 全体の CI をスキップしうるため。この場合は、**コミット本文にその理由を 1 行残す**（下の例）。
- 上記以外（今回のコミットが app・ビルド系を含む、または混在）→ `[skip ci]` は付けない。

まとめると、**「今回の対象に app・ビルド系が無い」かつ「既存の未 push コミットにも app・ビルド系が無い」→ 付ける。どちらかに app・ビルド系があれば付けない。**

`[skip ci]` を抑止した場合の本文注記の例（実在コミット `5cb7703` に倣う）:

```
未 push コミットに app 配下の変更が含まれるため、規約に従い [skip ci] は付けない。
```

### 4. コミットメッセージを作る（日本語・CLAUDE.md Git 規約）

- **件名は必ず日本語。** 変更を端的に表す。
- **本文（依頼に基づく変更の場合）**: 何を変えたかだけでなく、**どんな依頼で・なぜそう変えたか（意図・背景）**を日本語で数行残す。件名だけで意図が伝わるなら本文は省いてよい。
- `[skip ci]` を付ける場合は**件名の末尾**に置く（例: `景観色彩ガイドの用語を統一 [skip ci]`）。

#### `.svx`（記事ページ）の件名規約

対象に `.svx` が含まれる場合、frontmatter の `title:` から記事タイトルを取り、次のいずれかにする。

- **AI 初稿**（このセッションで雛形→本文を新規に書き起こした場合）: `記事タイトル [ai-draft]`
- **初稿ではないが AI の修正のみ**: `記事タイトル：修正内容 [ai-draft]`
- **人間が依頼した修正 / `author-style-writer` の編集モードで指示された修正**: `記事タイトル：依頼された修正内容`（`[ai-draft]` は付けない。依頼であることが分かるように書き、本文に依頼・意図を残す）
  - 例: `景観色彩の考え方：依頼に基づき導入文を追記`

初稿かどうかは、そのコミット前まで `draft` の雛形で本文が無く、今セッションで本文を書き起こしたなら「初稿」と判断する。

> 記事とそれ以外（コンポーネント・YAML 等）を 1 回の作業で触った場合、記事の公開・執筆が主眼なら記事タイトル基準の件名にし、本文でその他の変更に触れる。分けた方が履歴が読みやすいなら**コミットを分割**してよい（記事＝`.svx` と、それ以外を別コミットにする等）。分割時は各コミットで手順 2〜4 を個別に適用する。

#### 末尾のトレーラ

このリポジトリの慣例に従い、コミットメッセージの末尾に次の 1 行を付ける（本文がある場合は 1 行空けてから）。

```
Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
```

### 5. コミットする

1. **品質チェック（必要な場合）**: 対象に型チェックが有効なファイル（`.ts` / `.js` / `.svelte`。ただし `.svelte` の CSS のみ変更は除く）が含まれ、このセッションでまだ `npm run check` を通していない場合は、コミット前に `app` ディレクトリで実行し、エラー・警告を解消する。`css` / `svx` / `md` のみのコミットなら不要。
2. **明示パスでステージ**する。

   ```sh
   git add -- <path1> <path2> ...
   ```

3. **コミット**する。本文・改行・日本語を保つため、メッセージはスクラッチパッドの一時ファイルに書き出し、`-F` で渡す。

   ```sh
   git commit -F <一時ファイルのパス>
   ```

   件名だけで本文が無い場合は `-m "<件名>"` でよい。
4. `push` はしない。

### 6. 報告する

- コミット対象のファイル一覧（app・ビルド系 / ドキュメント・スキル系の分類つき）。
- `[skip ci]` を付けたか／付けなかったかと、その理由（未 push コミットの状況）。
- コミットメッセージ（件名）とコミットハッシュ。`git log -1 --stat` で確認結果を示す。
- **コミットに含めなかった dirty ファイル**があれば列挙し、「セッション外の変更のため対象外にした」旨を添える。

## 注意事項

- **push しない。** push は別途ユーザーの明示指示があってから。
- `git add -A` / `git add .` / `git commit -a` は禁止。常にパス明示。
- **セッション外の未コミット変更を勝手にコミットしない。** 迷ったら対象リストをユーザーに確認する。
- `[skip ci]` の判断は必ず**未 push コミット全体**（`git log HEAD --not --remotes`）を見て行う。今回のコミットがドキュメント・スキルのみでも、未 push に app 変更があれば付けない。
- コミットの分割が読みやすさに資する場合（記事の公開と無関係な雑多な変更が混ざる等）は分割を提案・実施してよい。
