# 記事 執筆スタイルガイド

次の3種類の解説記事の本文を執筆するときに従う、共通のスタイルガイドです。

- **CG・画像処理**（`app/src/routes/cg/**/<slug>/+page.svx`）
- **色の理論**（`app/src/routes/color-theory/<slug>/+page.svx`）
- **色の活用分野**（`app/src/routes/color-fields/<slug>/+page.svx`）

`content-ghostwriter` エージェント（本文の初回執筆）と `content-editor` エージェント（既存本文の編集）は、本文を書く前に必ずこのガイドを読み、すべてのルールに従ってください。

記事の種類によって異なるのは**大見出しに付ける「分類タグ」と、記事冒頭の「導入文」のテイスト**（いずれも後述）だけで、それ以外の文体・ディレクティブ・数式・コードのルールは3種類すべてで共通です。

このガイドは大きく2部構成です。

- **第1部（必須ルール）**: 本プロジェクト独自のMarkdown拡張（ディレクティブ）の使い方。
- **第2部（文章の型）**: 既存の `+page.svx` から抽出した、文体・見出し・段落・数式の書き方。

---

## 第1部 — 必須の執筆ルール（ディレクティブ）

このプロジェクトの `.svx`（mdsvex）は、通常のMarkdownに加えて独自のディレクティブを使います。
本文中で意味のある要素は、素のMarkdownではなく次のディレクティブで表現してください。

### ルール1: 重要な単語は `:Mark[]` で囲む

その記事で**初出・定義される専門用語**や、**読者に覚えてほしいキーワード**は `:Mark[...]` で囲みます。

```
区画化された画素の最小単位を:Mark[画素]といいます。
```

`:Mark[]` で囲んだ語は、通常表示では強調されますが、**暗記モード（anki mode）では隠されます**。つまり記事は、暗記モードで**穴埋め問題集**として機能します。この性質を前提に、次のルールを守ってください。

- **隠れても文脈から答えを復元できる語にだけ付ける**。穴埋めとして成立しない（前後の文脈から思い出せない）語に付けてはいけません。
- **乱用しない**。1文・1段落に空欄が多すぎると、空欄どうしが依存し合って解けなくなります。「その文・その節の主役になる、覚える価値のある語」だけに絞ります。普通名詞・接続表現・自明な語には付けません。
- **同じ語は原則として初出（1回目）だけ囲む**。2回目以降の登場はプレーンにします（同じ答えの空欄を繰り返さないため）。
- 用語を「〜といいます」「〜と呼ばれます」と定義する文では、定義される語を `:Mark[]` で囲むのが基本パターンです。
- 英略語などは、読み下しと略語の両方を囲んでよい（例: `:Mark[標本化]`、`:Mark[AD変換]`）。これは1つの定義に属するセットなので「同じ語の繰り返し」には当たりません。

#### `:Mark[]` と インラインコード `` `code` `` の使い分け

- `:Mark[]` … 概念・用語そのもの（読者に印象づけたいキーワード）。
- `` `code` `` … 記号・値・表記そのものを**リテラルとして引用**するとき（色コード・座標・数値表記など）。

```
トーンが`v`（vivid）の色は:Mark[`v2`]のように表します。
```

#### 数字は必ずインラインコードにする

**単体で成り立つかどうかに関わらず、数字は必ずインラインコード `` `code` `` として記述します。**

- `0と1` など、**数値が単体で現れる場合**は、`0` と `1` をそれぞれインラインコードにします。
- `2進数` や `3つの` など、**単語中の数値**も、数字の部分だけをインラインコードにします（`` `2`進数 ``、`` `3`つの ``）。
- `1mm` など、**単位つきの数値**は、単位も含めてインラインコードにします。
- `sin` など、**数学で定義されている関数名**もインラインコードにします。

なお、`:Mark[]` の中ではインラインコードは使えないので注意してください。

#### インラインコードの前後には空白を置かない

インラインコードと地の文の境界には、半角スペースを入れません。

- 悪い例: `輝度 `255` の白`
- 良い例: `輝度`255`の白`

### ルール2: 流れに乗せづらい補足情報は `:::Note` に書く

本筋の説明には含めたいが、文章の流れを止めてしまう**補足・脚注的な情報**は `:::Note` ブロックに切り出します。

```
:::Note
中間色相を表す記号は、両隣の原色の頭文字を:Mark[反時計]回りに組み合わせたものになっています。
:::
```

- 「なぜそうなっているか」「細かい例外」「記法の由来」など、知らなくても本筋は追えるが、あると理解が深まる情報に使います。
- `:::Note` の中でも `:Mark[]` や数式は通常どおり使えます。

### ルール3: 流れに乗せづらい具体例は `:::Example` に書く

本筋の抽象的な説明に対する**具体例**で、地の文に挿入すると流れが途切れるものは `:::Example` ブロックに切り出します。

```
:::Example
色相が`5R`、明度が`3`、彩度が`8`の色は、:Mark[5R 3/8]と表され、「5R、3の8」と読みます。
:::
```

- 「たとえば〜」で始まる具体例を独立させたいときに使います。
- 直前の抽象的な説明と1対1で対応させると効果的です。

### その他の利用可能なディレクティブ

`guide-content` レイアウトで安全に使えるディレクティブは次のとおりです。**草稿段階では `:Mark[]` / `:::Note` / `:::Example` を中心に**使い、以下は必要なときだけ使います。

| 記法 | 用途 |
| --- | --- |
| `:::Action` | 直後に置いたインタラクティブなデモの操作方法を促す一文（例: 「ドラッグで回転させてみよう」） |
| `::::CardGrid` + `:::TermCard{title="..." icon="1"}` | 並列に並ぶ複数の概念をカードで見せる |

#### ディレクティブのネスト規則

- コンテナディレクティブは `:::Name` ～ `:::` で囲みます。
- **入れ子にするときは、外側のコロンを1つ増やす**（外側 `::::`、内側 `:::`）。

```
::::CardGrid
:::TermCard{title="5つの原色" icon="1"}
（カードの中身）
:::
:::TermCard{title="中間色相を加えた10色相" icon="2"}
（カードの中身）
:::
::::
```

### やってはいけないこと

- **ボールド体（`**足し算**` のような `**...**`）は使わない**。強調したい語は素のボールドではなく、本プロジェクトの強調手段である `:Mark[]`（覚えてほしい用語・キーワード）で表現します。`:Mark[]` の対象にならない語は、強調せずプレーンな地の文のままにします（文の構成や接続表現で自然に伝わるように書く）。
- **存在しないデモコンポーネントを `import` しない**。`<script>` で `$lib/demo/...` を import してよいのは、そのコンポーネントが実在するときだけです。実在しないコンポーネントを import・参照するとビルド（`npm run check`）が壊れます。
- 図解・インタラクティブデモが欲しい箇所は、勝手にコンポーネントを作らず、`:::Note` で「ここに〇〇の図解を入れたい」と明記するか、`<!-- TODO: 〇〇の図解 -->` というHTMLコメントでプレースホルダを置いてください。
- `# 見出し`（H1）は使わない。H1はフロントマターの `title` から自動生成されます。

## 第2部 — 文章の型（既存記事から抽出）

### 文体の特徴

- **敬体（です・ます調）** で統一する。断定は「〜です」「〜ます」、説明は「〜になります」「〜といいます」。
- **教科書・解説調**。読者に語りかけすぎず、しかし硬すぎない、落ち着いた説明文。
- **教科書調でありながら、話題の自然な展開・流れを意識し、ストーリー感のある文章になるよう心がける。** 用語や事実を断片的に並べるのではなく、「なぜその話をするのか → どういうことか → だから次に何が言えるか」と話が一本につながっていくように書く。
  - 各節・各段落は、前の内容を受けて書き起こし、次の内容へ橋渡しして締めくくる。唐突に新しい話題を始めない。
  - 「疑問・課題を提示 → それに答える」「身近な現象 → その仕組み → 一般化」のように、読者が自然に納得しながら読み進められる順序で構成する。
  - 節の終わりに、その節で分かったことを次の節や関連ページへつなぐ一文を置くと流れが生まれる（例: 「このことは〜の話題に繋がっていきます」）。
- **定義 → 補足説明** の順で書く。まず用語や結論を提示し（「〜を:Mark[X]といいます。」）、続く文でその理由・仕組み・具体像を述べる。
- 1文には1つの内容。長い修飾の入れ子を避け、短め〜中程度の文を積み重ねる。
- 接続表現を効果的に使う: `つまり`（言い換え）、`たとえば`（例示）、`そのため`／`このことから`（帰結）、`また`（追加）、`しかし`（逆接）。
- 引用・原則の提示には Markdownの引用 `>` を使う（法則や定義の「原文」を示すとき）。

```
> 互いに独立した3つの色刺激を適切な比率で加法混色することによって、すべての色をつくり出すことができる
```

### 記事の冒頭（導入文）

記事は**いきなり最初の見出し（`##`）から始めない**。本文の先頭に、見出しの付かない**導入文**を置く。

**導入文のテイストは記事の種類によって変える。** CG・画像処理ページと、色の理論・色の活用分野ページとで、導入文の役割が異なる。どちらの場合も**リスト・箇条書き・ディレクティブは使わず、文章（段落）だけ**で構成する。

#### CG・画像処理ページの導入文（`routes/cg/`）

**読者に流れを見せ、読者を案内する**のが導入文の役割。

- 他の記事（特に一覧上の**前の記事**）とのつながりや、今回の記事のテーマに入る前の前置き・動機づけを、地の文で書く。
  - 他の記事とのつながりを示すときは内部リンクで繋ぐ。**リンクテキストはその記事のタイトルにする**（「前のページ」などにしない。詳細は「リンク・相互参照」を参照）。
- **3行程度に留める**。短く、読者を本題へ橋渡しするだけにする。
- どうしても導入として置きたい内容が長くなる場合は、最初の見出しの前で完結させようとせず、`##` 見出しを立てて節として書く（導入文そのものは3行程度を守る）。

#### 色の理論・色の活用分野ページの導入文（`routes/color-theory/` / `routes/color-fields/`）

読者に流れを案内するのではなく、**今回の主題となる概念を一言で説明する**のが導入文の役割。

- **1〜2行程度に留める**。
- 他の記事とのつながりや、今回の記事で説明することの要約にはしない。
  - **「〜の記事では、〜を説明しました」「この記事では、〜について見ていきます」といった前置きは禁止**。
- 主題そのものを端的に言い切る。次の例のように、対象を定義し直して読者の視点を引き上げる。

  ```
  ファッションコーディネートは、単に「服を組み合わせること」ではありません。身体をどう見せるか、どんな空気感を作るか、どんな人物像を表現するかをデザインすることです。
  ```

### 見出しの書き方

- 記事は `##`（大見出し＝節）と `###`（中見出し＝小節）で構成する。`####` はほとんど使わない。
- 見出しは **簡潔な名詞句** にする（「〜について」などは避け、内容を端的に表す語にする）。
- **すべての大見出し（`##`）には、記事の分類に対応する「分類タグ」ディレクティブを必ず付ける。** 使うタグは記事の種類によって2種類に分かれ、**フロントマターの分類キー（`group` か `grades` か）で判別する**。

  | 記事の種類 | ルート | フロントマターの分類キー | 大見出しに付けるタグ |
  | --- | --- | --- | --- |
  | CG・画像処理 | `routes/cg/` | `group`（分野） | `:WithGroupTag[見出し]{group="{['CG', 'ImgP']}"}` |
  | 色の理論 | `routes/color-theory/` | `grades`（級） | `:WithGradeTag[見出し]{grades="2,uc"}` |
  | 色の活用分野 | `routes/color-fields/` | `grades`（級） | `:WithGradeTag[見出し]{grades="2,uc"}` |

  フロントマターに `group` があれば `:WithGroupTag` を、`grades` があれば `:WithGradeTag` を使う。**2つのタグを取り違えたり、1つの記事で混在させたりしない。**

#### 分野タグ `:WithGroupTag`（CG・画像処理ページ）

  ```
  ## :WithGroupTag[画像のデジタル化]{group="{['CG', 'ImgP']}"}
  ```

  - `group` 属性は **Svelte 式として配列を渡す**。`group="{['CG', 'ImgP']}"` のように、ダブルクォートの中にマスタッシュ `{[...]}` を書く（mdsvex がこれを配列 props として解釈する）。CSV 文字列ではない。
  - 値には、**その節が対応する分野**の配列を書く。要素は `'CG'` / `'ImgP'` のみ。それ以外を書くと型エラー（`npm run check` で検出）になる。

#### 級タグ `:WithGradeTag`（色の理論・色の活用分野ページ）

  ```
  ## :WithGradeTag[等色実験]{grades="2,uc"}
  ```

  - `grades` 属性は **CSV 文字列**（級を `,` で連結したもの）を渡す。`:WithGroupTag` の Svelte 式配列とは記法が異なるので注意する（例: 級が1つなら `grades="2"`、複数なら `grades="3,2"`・`grades="2,uc"`）。
  - 値には、**その節が対応する級**を書く。要素は `'1'` / `'2'` / `'3'` / `'uc'` など、そのページのフロントマター `grades` に含まれる級。

#### 共通ルール（どちらの分類タグでも守る）

  - **その節が対応する分類だけ**を書く。ページ全体の分類（フロントマター）の部分集合でよく、節の内容に合わせて指定する（例: 画像処理だけの節は `group="{['ImgP']}"`、級`1`だけの節は `grades="1"`）。
  - 内容ごとに分けるのが難しい・分ける必要がない場合は、**ページのフロントマターの分類をそのまま全見出しに付ける**（CGなら `group` 全体、色記事なら `grades` 全体のCSV）。
  - **本文中の各見出しタグに現れた分類の和集合が、フロントマターの分類と一致している状態にする。**
    - 例（CG）: 文中に `## :WithGroupTag[タイトル]{group="{['CG', 'ImgP']}"}` という見出しを追加した場合、フロントマターを `group: ["CG", "ImgP"]` にする。
    - 例（色記事）: 文中の見出しが `grades="3"` と `grades="3,uc"` の2つなら、フロントマターを `grades: ["3", "uc"]`（和集合）にする。
- `###` 中見出しは常にプレーンテキスト（例: `### 等色式`、`### 等明度面`）。

### 段落の分け方

- **意味のまとまりごとに段落を分け、段落の間は空行を1つ**あける。1段落はおおむね1〜3文。
- このプロジェクトは `remark-breaks` を有効にしているため、**段落内の改行（単一の改行）はそのまま `<br>` になる**。
  - 関係の深い文を「同じ段落・別の行」にしたいときは改行で並べる。
  - 話題が変わるときは空行をあけて段落を切る。
- 並列する項目（特徴の列挙、用語のセットなど）は **箇条書き** にする。手順は番号付きリスト（`1.`）にする。

```
- :Mark[色相]：赤・黄・緑・青・紫など、どの色みをもたせるか
- :Mark[トーン]：どんなイメージをもたせるか
```

### 数式の記述方法

数式は **KaTeX** でレンダリングされます。インライン・ブロックともに **`$$...$$`** で囲みます（`$...$` 単独のドル記号は数式になりません）。

#### インライン数式はなるべく使わず、インラインコードで書く

地の文に式を埋め込むときは、**なるべくインライン数式（`$$...$$`）を使わず、インラインコード `` `code` `` で記述します**。

- 例: `$$\frac{1}{2}$$` ではなく `1 / 2`、`$$2dx$$` ではなく `2dx` と書きます。
- **KaTeX でしか表記できない数学記号やギリシャ文字**（`$$\equiv$$`、`$$\times$$`、上付き・下付き文字、ギリシャ文字、`$$[R]$$` のような色記号の慣習表記など）が含まれる場合**のみ**、インライン数式にします。
  - たとえばギリシャ文字の `θ` は、地の文でもインラインコードにせず `$$\theta$$` と書きます。

- **インライン数式**（KaTeX が必要なときのみ）: 地の文の中に `$$...$$` を埋め込む。

  ```
  赤$$[R]$$・緑$$[G]$$・青$$[B]$$の`3`色の光を重ねます。
  ```

- **ブロック（中央寄せの独立した式）**: その**段落が数式 `$$...$$` だけ**で構成されるようにする（前後を空行で挟み、`$$` を独立した行に置く）。

  ```
  $$
  [X] \equiv R[R] + G[G] + B[B]
  $$
  ```

- **複数行の式** は `\begin{align*} ... \end{align*}` を使い、**行末の改行は `\\\\`（バックスラッシュ4つ）** で表す（mdsvex のエスケープを通すため2つではなく4つ必要）。

  ```
  $$
  \begin{align*}
  [X] &\equiv R_1[R] + G_1[G] + B_1[B] \\\\
  [Y] &\equiv R_2[R] + G_2[G] + B_2[B]
  \end{align*}
  $$
  ```

- **`\frac` による分数を含む数式ブロック** では、`\\\\`（バックスラッシュ4つ）で改行すると分数の高さの分だけ行が詰まって見えます。この場合は**行末の改行を `\\\\\\\\`（バックスラッシュ8つ）** にして行間を確保します。

  ```
  $$
  \begin{align*}
  e + \frac{dy}{dx} &> \frac{1}{2} \\\\\\\\
  2dx \cdot e + 2dy &> dx \\\\\\\\
  2dx \cdot e + 2dy - dx &> 0
  \end{align*}
  $$
  ```

- **プライム記号（`'`）は使わず `^{\prime}` と書く**。変換後の座標などを表す `x'`・`y'` を、アポストロフィ `'`（`x'`）で書いてはいけません。mdsvex の smartypants がレンダリング前にこの `'` をカーリークォート `’`（U+2019）に変換してしまい、KaTeX が「LaTeX-incompatible input ... Unrecognized Unicode character "’" (8217) [unknownSymbol]」という警告を出すためです。`^{\prime}` と書けばクォート文字を含まないので変換を受けず、表示結果は `x'` と同じになります。

  ```
  $$
  \begin{pmatrix} x^{\prime} \\\\ y^{\prime} \end{pmatrix} = \begin{pmatrix} \cos\theta & -\sin\theta \\\\ \sin\theta & \cos\theta \end{pmatrix} \begin{pmatrix} x \\\\ y \end{pmatrix}
  $$
  ```

#### 同じ対象を表す記号は、インライン数式とインラインコードを混在させない

ある数学的対象を表す記号は、記事全体を通して**インライン数式（`$$...$$`）かインラインコード `` `code` `` のどちらか一方に統一**します。同じ対象を、ある箇所ではインライン数式、別の箇所ではインラインコードで書いてはいけません。

- たとえば、行列をインライン数式で表記する箇所が1つでもあれば、その行列を表す他の箇所もすべてインライン数式で書きます。
- 悪い例: ``$$T^{-1}$$は`T`の:Mark[逆行列]で``
- 良い例: `$$T^{-1}$$は$$T$$の:Mark[逆行列]で`

- **色・記号の慣習**（色彩分野の記事に合わせる）:
  - 色光・色刺激は角括弧つきの大文字で書く: `$$[R]$$`、`$$[G]$$`、`$$[B]$$`、`$$[X]$$`。
  - 混色量・三刺激値は角括弧なしの大文字で書く: `$$R, G, B$$`。
  - 「等色（見た目が一致）」を表す記号は等号 `=` ではなく `$$\equiv$$` を使う。
  - 掛け算は `$$\times$$`、その他は標準のLaTeX記法。

### コードブロック・擬似コードの書き方

アルゴリズムの手順は、特定の言語のソースコードではなく**擬似コード**で説明します。

#### 擬似コードの規約

- **アルゴリズムは擬似コードで説明する**（実在の言語の正確な構文に縛られず、手順が読み取れることを優先する）。
- コードブロックの**言語名（フェンスの言語指定）は `js`** とする。
- **代入は矢印 `<-`** で表記する（`=` は使わない。例: `y <- y_start`）。
- **コメントは `// コメント`** で表記する。
- 条件分岐・繰り返しは行末にコロン `:` を付けて表す（例: `for x = x_start to x_end:`、`if d > 0:`、`else:`）。
- 変数名はスネークケースにする（例: `x_start`、`y_end`）。
- 関数呼び出しは `関数名(引数)` の形で書く（例: `plot(x, y)`）。
- 演算子（`+` `-` `*` `>` など）の前後には半角スペースを入れる（例: `d <- 2 * dy - dx`）。

#### コードブロックの体裁

- **インデントは半角スペース `4` つで統一する**（タブやスペース `2` つを混在させない）。
- **行末にコメントを置くときは、コードとコメントの間に半角スペースを `2` つ入れる**（例: `e <- 0  // 画素中心からの誤差`）。

例:

````
```js
e <- 0  // 画素中心からの誤差
y <- y_start
for x = x_start to x_end:
    plot(x, y)  // 画素 (x, y) を塗る
    e <- e + a  // a = dy / dx（傾き）
    if e > 0.5:
        y <- y + 1
        e <- e - 1
```
````

### リンク・相互参照

- 関連する既存ページへは**内部リンク**で繋ぐ。パスは**末尾スラッシュ付き**で書く。

  ```
  このことは[RGB表色系](/color-theory/rgb-color-system/)の話題に繋がっていきます。
  ```

- 記事どうしの関連が深い箇所では、CG・画像処理／色の理論／色の活用分野のどれであっても積極的に相互リンクする。
- **記事そのものを参照するとき（記事全体へ読者を誘導するとき）は、リンクテキストをその記事のタイトルにする**。「前のページ」「次の記事」のような相対的・指示的な語をリンクテキストにしない。
  - 悪い例: `[前のページ](/cg/basics/camera-capture-and-cg/)では、`
  - 良い例: `[カメラでの撮影とCG](/cg/basics/camera-capture-and-cg/)では、`
- **ある用語を他の記事で詳しく説明している場合は、その用語自体をリンクにする**。文章の流れを損なわないよう、リンクテキストを他記事のタイトルにせず、地の文に出てくる**用語そのもの**をリンクテキストにする。
  - 悪い例: `光の三原色については[RGB表色系](/color-theory/rgb-color-system/)で説明しています。`
  - 良い例: `[光の三原色](/color-theory/rgb-color-system/)を適切な比率で混ぜることで、`
