# 仕様壁打ちスキル

新機能の**実装に着手する前**に、まだ曖昧な仕様をユーザーと**対話で壁打ちして固める**ためのスキルです。

このリポジトリのワークフロー上は「アイデア → **【仕様の壁打ち＝本スキル】** → 実装」に位置します。

**重要な方針:**

- **壁打ちの主目的は仕様を固めること。** 固めた仕様は最後に `spec/` 配下の Markdown ファイルとして書き出す（手順3）。
- **実装に飛ばない。** コードの変更・アプリのファイル作成は行わない。仕様が固まって `spec/` に書き出すまでが本スキルの役割。
- **勝手に決めない。** 仕様の決定権はユーザーにある。AI は引き出し役・網羅役・批判役に徹し、決定は都度ユーザーに確認する。

## 入力

- **機能名**（必須・引数）: 壁打ちの対象とする新機能の名前。
  - 例: `/spec-sparring トーン別配色のお気に入り保存`
  - 引数が無ければ、まず「どんな機能を壁打ちしたいか（機能名・一言説明）」をユーザーに尋ねる。

## 進め方（壁打ちの流儀）

進め方には3つの役割があり、これらを同時に担う。

1. **引き出す（主軸）** — ユーザーが頭の中に持っている構想を、質問で引き出すことを最優先にする。AI が先に答えを埋めるのではなく、ユーザーに考えさせ・言語化させる。ユーザー主導。
2. **網羅する（AI の責務）** — 下の「論点チェックリスト」を使い、抜け漏れている観点に気づいたら必ず質問として持ち出す。ユーザーが触れていない論点を素通りさせない。
3. **突っ込む（AI の責務）** — 各論点で、前提を鵜呑みにせず疑う。代替案・トレードオフ・見落としがちなリスクやエッジケースを積極的に提示し、「壁」として打ち返す。ただし否定のための否定はせず、より良い仕様に収束させるための問いかけとして行う。

### 対話の作法

- **一度に大量に聞かない。** 1回のやり取りでは1つの論点クラスタ（数問程度）に絞り、テンポよく往復する。質問を箇条書きで羅列して丸投げしない。
- **合意を明示する。** 各論点で結論が出たら「ここはこう決まりましたね」と要点を短く確認してから次へ進む。
- **未確定は持ち越す。** その場で決められない論点は「保留」として控え、無理に埋めない。書き出す仕様でも「未確定・保留事項」として明示する。
- **用語を揃える。** 既存のコードや `spec/` の既存仕様で使われている用語（PCCS の色相・トーン表記など）に合わせて会話する。

## 手順

### 1. 事前調査（会話の土台づくり）

壁打ちを始める前に、機能名を手がかりに軽くコードベースを調べ、的外れな質問を避ける。

- `spec/` に既存の仕様ファイルがあれば読み、関連する機能・決定事項・用語を把握する。
- 関連しそうな既存機能・ページ・コンポーネントを把握する（`app/src/routes/`, `app/src/lib/`）。重複・連携・整合の観点を用意する。
- 機能が触れるドメイン（PCCS の色相・トーン、色距離、配色技法、イメージ別配色など）を、関連コードや既存仕様から把握する。
- 技術スタック上の制約（Svelte / Threlte 等）を、関連コードから把握する。

調査で分かったこと・関連しそうな既存機能を1〜2文でユーザーに共有し、「この理解で合っていますか」と確認してから壁打ちに入る。

### 2. 壁打ちループ

論点チェックリストを上から順に（ただし会話の自然な流れを優先して）扱う。各論点で:

1. その論点についてユーザーに問いかけ、考えを引き出す（役割1）。
2. 出てきた案の前提・抜け・リスクを突っ込み、必要なら代替案やトレードオフを示す（役割3）。
3. ユーザーが触れていない関連観点があれば追加で問う（役割2）。
4. 結論が出たら要点を確認して合意し、次の論点へ。決められなければ保留にする。

#### 論点チェックリスト

- **目的・背景** — なぜこの機能が必要か。誰の・どんな課題を解決するのか。
- **スコープ** — 今回やること／やらないこと（初回リリースの線引き）。
- **ユーザーストーリー・主要ユースケース** — 誰が、どんな状況で、どう使うか。
- **機能要件** — 入力 → 処理 → 出力の流れ、UI・操作、画面遷移。
- **既存機能・ページとの関係** — 重複していないか、どこと連携するか、一貫性は取れるか。
- **ドメインルールとの整合** — PCCS の色相・トーン、色距離、配色技法、イメージ別配色などのルールと矛盾しないか。
- **データ・状態** — 扱うデータの構造（色データ JSON 等）、状態管理、永続化の要否。
- **エッジケース・異常系** — 無彩色・境界値・空入力・エラー時の挙動など。
- **非機能要件** — アクセシビリティ（a11y）、モバイル対応、パフォーマンス、必要なら SEO。
- **制約** — 技術スタック（Svelte / Threlte 等）や既存アーキテクチャ上の制約、後方互換。
- **受け入れ条件** — 「完成」をどう検証するか（動作の合否基準）。

（この機能に無関係な論点は飛ばしてよい。逆に、この機能特有の論点があれば柔軟に追加する。）

### 3. 仕上げ（仕様の書き出し）

主要な論点に区切りがついたら、合意した内容を構造化して**まず会話上に**まとめて提示し、「この内容で仕様として固めてよいか」を確認する。修正要望があればループに戻る。

合意が取れたら、`spec/` 配下に Markdown ファイルとして書き出す。

- **保存先**: `spec/<スラッグ>.md`
  - スラッグは機能名から英語の kebab-case を作る（例: 「トーン別配色のお気に入り保存」→ `tone-palette-favorites`）。作成前にファイルパスをユーザーに確認する。
  - `spec/` ディレクトリが無ければ作成する。同名ファイルがあれば上書きせず、内容を更新してよいかユーザーに確認する。
- **記載項目**（会話で固まった範囲で記述する。埋まらない項目は「未確定」と明記）:
  - `# <機能名>`
  - `## 目的・背景`
  - `## スコープ`（やること／やらないこと）
  - `## ユーザーストーリー・主要ユースケース`
  - `## 機能要件`
  - `## 既存機能との関係・整合`
  - `## ドメインルール上の考慮`
  - `## データ・状態`
  - `## エッジケース・異常系`
  - `## 非機能要件`（a11y／モバイル／パフォーマンス 等）
  - `## 制約`
  - `## 受け入れ条件`
  - `## 未確定・保留事項`

ファイルを書き出したらパスを伝えて完了。**実装着手はしない**（ユーザーが明示的に次工程を求めた場合のみ、別途対応する）。

## 注意事項

- `docs/` や `.steering/` は参照・生成しない（いずれも廃止予定）。仕様の成果物は `spec/` 配下の Markdown のみ。
- コードの変更・アプリのファイル作成など実装は行わない。仕様を固めて `spec/` に書き出すまでが役割。
- 仕様は勝手に確定させず、決定は都度ユーザーに確認する。曖昧な点・推測で埋めた点は「未確定・保留事項」として明示する。
- 質問は小分けにし、対話のテンポを保つ。論点を一度に羅列して丸投げしない。
- 用語は既存コード・既存仕様に合わせる。
