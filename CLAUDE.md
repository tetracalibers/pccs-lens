# CLAUDE.md (プロジェクトメモリ)

## 概要

開発を進めるうえで遵守すべき標準ルールを定義します。

## プロジェクト構造

### ドキュメントの分類

#### 永続的ドキュメント（`docs/`）

アプリケーション全体の基本設計や方針を定義するドキュメント。

- **architecture.md** - 技術仕様書
  - テクノロジースタック
  - アーキテクチャ方針
  - データの扱い
  - 開発ツールと手法
  - コーディング規約の概要

- **repository-structure.md** - リポジトリ構造定義書
  - フォルダ・ファイル構成
  - ディレクトリの役割
  - ファイル配置ルール

- **development-guidelines.md** - 開発ガイドライン
  - コーディング規約
  - 命名規則
  - フォーマット規約
  - スタイリング規約
  - テスト規約
  - Git規約

#### ドメインドキュメント（`docs/domains/`）

アプリケーション特有の用語や前提とする専門知識を定義するドキュメント。

- **glossary.md** - ユビキタス言語定義
  - 色の理論に関する用語
  - アプリケーション内の用語やUI用語（コード上での命名との対応）

- **color-analysis-rules.md** - 色分析ルール定義
  - 色分析ルール定義書
  - 色距離の計算方法
  - PCCSの色相・トーンの構造や表記
  - 配色技法の定義と判定条件

- **image-based-color-rules.md** - イメージ別配色ルール定義書
  - F3で使う配色の説明やサジェストの詳細ルール

#### 仕様ドキュメント（`spec/`）

新機能や仕様変更について、実装に着手する前に `spec-sparring` スキルで対話的に固めた仕様の書き出し先。機能単位で1ファイルを作成し、実装の拠り所とする。

- **spec/[スラッグ].md** - 固めた仕様
  - 目的・背景 / スコープ（やること・やらないこと）
  - ユーザーストーリー・主要ユースケース
  - 機能要件 / 既存機能との関係・整合 / ドメインルール上の考慮
  - データ・状態 / エッジケース・異常系
  - 非機能要件 / 制約 / 受け入れ条件
  - 未確定・保留事項

## 開発プロセス

### 機能追加・修正時の手順

#### 1. 仕様の壁打ち（`spec-sparring`）

実装に着手する前に、`/spec-sparring [機能名]` で仕様を対話的に固める。

- ユーザーへの質問で構想を引き出すことを主軸に、論点（目的・スコープ・ユーザーストーリー・機能要件・既存機能との整合・ドメインルール・データ／状態・エッジケース・非機能要件・制約・受け入れ条件）を体系的に網羅する。
- 各論点で前提を疑い、代替案・トレードオフ・リスクを提示して「壁打ち」する。
- **勝手に決めない。** 決定は都度ユーザーに確認し、曖昧な点は「未確定・保留事項」として明示する。
- 合意が取れたら仕様を `spec/[スラッグ].md` に書き出す。ここまでが本工程で、**実装には飛ばない**。

#### 2. 実装

`spec/[スラッグ].md` に固めた仕様に基づいて実装を進める。

#### 3. ドキュメント更新（必要な場合のみ）

今回の作業により、ドキュメント（`docs/`）の内容と実際の実装やディレクトリ構成に矛盾が生じる場合や、`docs/domains` に追記すべきドメイン用語がある場合は、ドキュメントの更新を行う。

## ドキュメント管理の原則

### 永続的ドキュメント（`docs/`）
- アプリケーションの基本設計を記述
- 頻繁に更新されない
- 大きな設計変更時のみ更新

### 仕様ドキュメント（`spec/`）
- 機能単位で `spec-sparring` の成果物として作成
- 実装の拠り所となる合意済み仕様を記録
- 未確定・保留事項を明示し、決定の経緯を残す

## 注意事項

- 実装前に `spec-sparring` で仕様を対話的に固め、各段階でユーザーの合意を得る
- 仕様は勝手に確定させず、推測で埋めた点・曖昧な点は「未確定・保留事項」として明示する
- 永続的ドキュメントと仕様ドキュメントを混同しない
- コード変更後は「品質チェック」節の基準に従ってリント・型チェックを実施する
- セキュリティを考慮したコーディング（XSS対策、入力バリデーションなど）

## Git規約

コミットメッセージは必ず日本語で記述する。

`.svx` ファイル（記事ページ）のコミットメッセージは、次の規約に従う。

- AIが書いた初稿の場合：`記事タイトル [ai-draft]`
- 初稿ではないが、AIによる修正しか含まない場合：`記事タイトル：修正内容 [ai-draft]`
- 人間が加えた修正のコミットを依頼された場合：`記事タイトル：修正内容`

ドキュメントとして扱われる（アプリの実行に関わらない）Markdownファイルのみの変更や、スキルの変更は、`[skip ci]` をつけてコミットする。ただし、未 push のコミットにアプリの実行やビルドに関わる変更が含まれている場合は、`[skip ci]` をつけない。

## 品質チェック

`app`ディレクトリ内で実装を行ったら、次のコマンドを実行し、エラーや警告が出た場合は修正を行う：

- `npm run check`

ただし、次の基準で実行の要否を判断する。

- ts・js・svelte ファイルなど、型チェックが有効なファイルを変更した場合のみ実行する
- css・svx・md ファイルなどの変更時は実行しない
- svelte ファイルの変更であっても、CSS のみ（`style` 内のみ）の変更であれば実行しない
